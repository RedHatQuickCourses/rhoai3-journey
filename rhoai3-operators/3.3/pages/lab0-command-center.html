Lab 0 — CLI-Driven Setup: The Terminal Interface

Before you build the factory, you need a control room. Leave the brittle laptop toolchain behind.

Reproducible Models-as-a-Service (MaaS) and GitOps demand a consistent shell. Clicking through OperatorHub is error-prone and lacks an audit trail. We will use the OpenShift Web Terminal Operator, which gives every authorized admin an authenticated oc session inside the console—meaning no VPN-split routing debates for basic tasks.

Step 1: The Bootstrapping Paradox

To run declarative commands, you need a client. Because we are establishing our baseline, we will perform one manual UI action to bootstrap our CLI environment.

  1. [cite_start]Log in to the OpenShift console as a cluster admin4

  2. Open Ecosystem → Software Catalog.

  3. Search for terminal and install the Web Terminal (Red Hat)4.

  4. Once installed, refresh the page and click the terminal icon (>_) in the console header (top right) to start your workspace.

You now have an authenticated oc session directly in your browser. Every command from here on is declarative.

Step 2: Phase A — Core Prerequisites (Trust, Hardware, & Scheduling)

Before we install the AI Factory manager, we must lay the foundational concrete. We will apply these subscriptions in order.

1. cert-manager Operator

Trust comes before traffic. KServe-oriented stacks commonly depend on in-cluster certificate issuance. Install the cert-manager Operator to handle TLS.

cat <<EOF | oc apply -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: openshift-cert-manager-operator
  namespace: cert-manager-operator
spec:
  channel: stable-v1
  name: openshift-cert-manager-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

2. Node Feature Discovery (NFD)

GPUs you cannot label are GPUs you cannot schedule reliably. We must install the Node Feature Discovery Operator to publish hardware labels on our nodes. Node Feature Discovery is required for scheduling NUMA-aware workloads.

cat <<EOF | oc apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: openshift-nfd
---
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: openshift-nfd
  namespace: openshift-nfd
spec:
  targetNamespaces:
  - openshift-nfd
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: nfd
  namespace: openshift-nfd
spec:
  channel: stable
  name: nfd
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

3. Hardware Accelerator Operator (NVIDIA)

Once NFD detects your hardware, you need the specific operator to manage the drivers and device plugins[cite: 4291]. For this MaaS use case, we are assuming NVIDIA GPUs, so we will install the NVIDIA GPU Operator version 24.3.

Platform Engineering Hygiene: Namespace Isolation

We are isolating the GPU operator in the nvidia-gpu-operator namespace. Furthermore, the operator will sit idle until we declare our intent by creating a ClusterPolicy Custom Resource to actually deploy the drivers to the worker nodes.

 
cat <<EOF | oc apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: nvidia-gpu-operator
---
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: nvidia-gpu-operator-group
  namespace: nvidia-gpu-operator
spec:
  targetNamespaces:
  - nvidia-gpu-operator
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: gpu-operator-certified
  namespace: nvidia-gpu-operator
spec:
  channel: v24.3
  name: gpu-operator-certified
  source: certified-operators
  sourceNamespace: openshift-marketplace
EOF

Wait a moment for the CSV to succeed, then apply the ClusterPolicy to activate the GPUs:

cat <<EOF | oc apply -f -
apiVersion: nvidia.com/v1
kind: ClusterPolicy
metadata:
  name: gpu-cluster-policy
spec: {}
EOF

4. Red Hat build of Kueue Operator

To unlock Hardware Profiles and GPU-as-a-Service capabilities (including fair-share scheduling), we must install the Red Hat build of Kueue Operator.

cat <<EOF | oc apply -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: kueue
  namespace: openshift-operators
spec:
  channel: stable
  name: kueue
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Step 3: Phase B — The Meta-Operator

Next, we deploy the factory manager. To use OpenShift AI 3.0, install the Red Hat OpenShift AI Operator on a cluster running OpenShift Container Platform 4.19 or later and select the fast-3.x channel.

Service Mesh 3 Auto-Management

In OpenShift AI 3.x, you do not need to manually install the Service Mesh 3 Operator unless configuring specific external components. The OpenShift AI meta-operator will automatically install and manage it when components like the Llama Stack are activated.

 
cat <<EOF | oc apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: redhat-ods-operator
---
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: rhods-operator
  namespace: redhat-ods-operator
spec:
  targetNamespaces:
  - redhat-ods-operator
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: rhods-operator
  namespace: redhat-ods-operator
spec:
  name: rhods-operator
  channel: fast-3.x
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Step 4: Phase C — Scale-Out Machinery (MaaS Dependencies)

An unmanaged endpoint is a liability. Before inference traffic flows, we must install the policy plane required for Phase 2 and 3 of our journey. To deploy models by using Distributed Inference with llm-d, you must install the Red Hat Connectivity Link Operator and the Red Hat Leader Worker Set Operator.

1. Red Hat Connectivity Link Operator

This installs the connectivity layer used with Gateway API patterns for governed MaaS exposure.

cat <<EOF | oc apply -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: connectivity-link-operator
  namespace: openshift-operators
spec:
  channel: stable-v1
  name: connectivity-link-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

2. Leader Worker Set Operator

This provides the orchestration primitive for leader/worker distributed pods. It lets very large models span devices and nodes.

cat <<EOF | oc apply -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: leader-worker-set
  namespace: openshift-operators
spec:
  channel: stable
  name: leader-worker-set
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Step 5: Verification

Wait for the Concrete to Dry

Do not proceed until the Operator Lifecycle Manager (OLM) has successfully resolved these subscriptions into running ClusterServiceVersions (CSVs). A stuck CSV is a P1 for the factory schedule.

Run the following commands in your Web Terminal to verify all components have successfully installed:

# Check cert-manager
oc get csv -n cert-manager-operator | grep cert-manager

# Check NFD
oc get csv -n openshift-nfd | grep nfd

# Check NVIDIA GPU Operator
oc get csv -n nvidia-gpu-operator | grep gpu-operator

# Check RHOAI Operator
oc get csv -n redhat-ods-operator | grep rhods-operator

# Check Connectivity Link & Leader Worker Set & Kueue
oc get csv -n openshift-operators | egrep "connectivity-link|leader-worker-set|kueue"

Expect the PHASE column to report Succeeded for all items before moving on.
