Lab 1 — Activating the AI Factory (DataScienceCluster)

Operators provide the potential. The DataScienceCluster provides your intent.

In Lab 0, we installed the meta-operator and all the necessary infrastructure operators (NFD, GPU, Kueue, cert-manager, Connectivity Link, and Leader Worker Set). However, if you look at your cluster right now, the Red Hat OpenShift AI (RHOAI) meta-operator is sitting entirely idle.

This is by design. The meta-operator acts as a factory manager waiting for your blueprint. You must declare which RHOAI subsystems should be activated. This is how you keep idle controllers from nibbling RAM and CPU in multi-tenant clusters.

1. The Strategy: Selective Activation

The Golden Rule of Platform Engineering applies here: Provision against outcomes. Do not turn every dial to "Managed." For our Models-as-a-Service (MaaS) factory path, we will use selective activation:

  • Dashboard — Managed (Provides the Model Catalog and user interface).

  • Model Registry — Managed (With a dedicated rhoai-model-registries namespace to act as your AI "Title Office").

  • KServe — Managed (The core serving infrastructure for large models).

  • Models-as-a-Service (MaaS) — Managed (Deploys the MaaS API, creates default service tiers, and sets up RBAC roles for access control).

  • Kueue — Managed (Controls resource allocation and queue management for AI workloads).

  • Unused Components — Removed (We will explicitly disable legacy ModelMesh, Ray, Codebases, and TrustyAI for this specific foundation phase to reduce reconciliation overhead and attack surface).

Schema Evolution Reality Check

The DataScienceCluster schema changes between releases. In RHOAI 3.3, the API version is v2. If an apply fails with unknown fields during future upgrades, run oc explain datasciencecluster.spec on your cluster and reconcile your manifest with the installed CRD.

2. Understanding the Kueue Configuration

Because we are building a factory that shares expensive GPU resources, queuing and quota management are critical. In our blueprint below, we are adding two specific configurations for Kueue:

  • spec.components.kueue.managementState: Managed

    • What this does: This tells the RHOAI meta-operator to actively deploy, configure, and manage the lifecycle of the Kueue integration components specific to OpenShift AI. Instead of forcing you to manually configure default ClusterQueue and LocalQueue objects, the meta-operator wires these up automatically so your data scientists can submit distributed workloads immediately.

  • spec.dashboardConfig.disableKueue: false

    • What this does: This toggle directly controls the user interface. Setting it to false enables the Kueue-related UI elements within the OpenShift AI Dashboard. This exposes hardware profiles, quota management, and workload tracking directly to your users, bridging the gap between the backend scheduling engine and the frontend data science experience.

3. Applying the Blueprint

We will use our Web Terminal to apply the DataScienceCluster Custom Resource.

In the spec.components section, setting a field to Managed tells the Operator to actively install and manage the component. Setting it to Removed tells the Operator to actively manage the component by ensuring it is not installed.

Copy and paste the following into your authenticated Web Terminal:

cat <<EOF | oc apply -f -
apiVersion: datasciencecluster.opendatahub.io/v2
kind: DataScienceCluster
metadata:
  name: default-dsc
spec:
  dashboardConfig:
    disableKueue: false
  components:
    dashboard:
      managementState: Managed
    modelregistry:
      managementState: Managed
      registriesNamespace: rhoai-model-registries
    kserve:
      managementState: Managed
      serving:
        managementState: Managed
        name: knative-serving
    modelsAsService:
      managementState: Managed
    kueue:
      managementState: Managed
      defaultClusterQueueName: default
      defaultLocalQueueName: default
    modelmeshserving:
      managementState: Removed
    codebases:
      managementState: Removed
    ray:
      managementState: Removed
    trustyai:
      managementState: Removed
EOF

4. Verification & Validation

Applying the CR is instantaneous, but the meta-operator needs time to orchestrate the deployment of these subsystems across your namespaces.

Check the DSC Phase

Run the following command to check the top-level status of the DataScienceCluster:

oc get datasciencecluster default-dsc -o jsonpath='{.status.phase}{"\n"}'

Target: Ready. (Note: During the initial installation, it might take several minutes for the status phase to change from Progressing to Ready).

Check the Component Pods

Confirm that the meta-operator has successfully spun up the pods for your activated components in the redhat-ods-applications namespace:

oc get pods -n redhat-ods-applications

You should see running pods for rhods-dashboard, kserve-controller-manager, Kueue controllers, and the various MaaS controllers.

Access the Dashboard

RHOAI commonly exposes the user experience via a gateway-style route. Let’s fetch the route to your newly minted AI Factory floor:

oc get route -n redhat-ods-applications | grep rhods-dashboard

Open that URL in your browser. Log in with your OpenShift credentials, and confirm that the navigation menu displays the Model Catalog, Model Registry, and Kueue-backed Hardware Profiles options.
