Architecting the Operator Foundation

Stop consuming tokens. Start producing them. But first, we need to configure the platform.

Enterprises default to a token-consumption posture: they rent intelligence from the public cloud, absorb unpredictable unit economics, and route sensitive prompts off-prem. Shadow AI makes it worse—ad-hoc clusters, hand-tuned pods, and GPUs purchased as team pets instead of shared, governed capacity. None of that maximizes your GPU ROI or shrinks the time-to-first-token (TTFT) for the workloads that matter to the business.

This course is Phase 0 of the Foundations to Scalable AI platform arc. The modules to follow assume you have a greenfield OpenShift AI 3.3 environment with GPUs.

Here, you act as the platform engineer who installs only the operators and operands the journey actually needs to build a Generative AI application platform that provides AI Model Inference at Scale using Models-as-a-Service (MaaS).

What is an Operator? (And why do we care?)

An Operator is a software extension that uses Custom Resources (CRs) to manage applications and their underlying components. Think of it as an automated Site Reliability Engineer (SRE) encoded into software. Operators are long-lived reconcilers. They continuously monitor the cluster state and actively reconcile the desired state you declare in your YAML against the actual state running on the execution plane.

Red Hat OpenShift AI uses a meta-operator approach. Installing the Red Hat OpenShift AI Operator does not instantly materialize every user interface and serving pod. Instead, this core meta-operator deploys most and maintains all underlying components and sub-operators. It acts as a factory manager waiting for a declarative blueprint—specifically, the DataScienceCluster Custom Resource—to tell it exactly what subsystems to activate.

Architectural Reality Check for RHOAI 3.3

OpenShift AI 3.0 and later introduces significant technology and component changes and is intended for new installations only. You cannot directly upgrade from OpenShift AI 2.25 or earlier.

To build our MaaS factory, plan to deploy models using Distributed Inference with llm-d. Your cluster must be running OpenShift version 4.20 or later. Distributed Inference with `llm-d is the high-performance, Kubernetes-native framework designed specifically for serving LLMs at scale.

Your Mission: Assemble the Platform

In this course, you will take on the role of the Platform Engineer to lay the groundwork for this entire journey. We will avoid local machine configurations entirely by utilizing native cloud tools.

You will execute the following workflow:

  1. Lab 0: Install the OpenShift Web Terminal Operator to establish a zero-dependency command line interface for our cluster.

  2. Lab 1: Deploy the foundational dependencies (cert-manager and Connectivity Link) required for Phase 2 serving and Phase 3 MaaS scaling.

  3. Lab 2: Install the core OpenShift AI Operator and map your physical hardware using NFD to support Phase 1.

  4. Lab 3: Configure the DataScienceCluster to activate the exact components needed for your token factory.

The Shift to Declarative Deployment

Throughout this course, we standardize on GitOps-friendly manifests and CLI-driven deployments. Clicking through the Ecosystem (OperatorHub) user interface is fine for a temporary sandbox, but reproducible Models-as-a-Service (MaaS) demands a consistent, auditable trail. We will declare our subscriptions, channels, and dependencies as code.
