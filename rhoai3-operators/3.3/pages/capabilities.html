Architecting the Operator Foundation

This page is the menu, not the meal. You are choosing control-plane services that make the execution plane (GPUs, serving pods, gateways) predictable, shared, and measurable.

To build a generative AI application platform that provides AI Model Inference at Scale using Models-as-a-Service (MaaS), you need a specific combination of infrastructure, serving, and governance operators. Provisioning every operator in the catalog is an anti-pattern: you pay in CPU, memory, reconciliation storms, and upgrade risk. Instead, we provision against specific factory outcomes.

Here is the roster of operators required to build your industrialized AI token platform.

1. Core & Infrastructure Operators (The Factory Floor)

These operators provide the foundational hardware detection and acceleration required for generative AI at scale.

  • Red Hat OpenShift AI Operator: The core meta-operator that manages the entire AI lifecycle platform. It deploys and maintains all components and sub-operators that are part of OpenShift AI. It acts as a factory manager waiting for a DataScienceCluster custom resource. To enable the MaaS architecture, you must install version 3.4 or later.

  • Node Feature Discovery (NFD) Operator: Publishes hardware labels on nodes, such as CPU flags, PCI devices, and accelerators. This ensures GPU operators and schedulers see the same truth, meaning placement and profiles stop being guesswork.

  • Hardware Accelerator Operators: Depending on your infrastructure, you must install the specific operator to manage the drivers and device plugins for your hardware. Supported options include the NVIDIA GPU Operator, AMD GPU Operator, Intel Gaudi Base Operator, or IBM Spyre Operator. This allows for higher utilization per dollar by sharing slices instead of dedicating one model per card.

2. Operators for Inference at Scale & MaaS (The Production Lines)

To deliver Large Language Models (LLMs) as a governed, shared resource across your enterprise, you must install the following operators to handle networking, security, and distributed serving.

  • Red Hat Connectivity Link Operator: This is critical for MaaS. It provides Kuadrant-style policy attachment, authN/Z hooks, and rate limits on ingress to models. This turns raw endpoints into products with SLAs—internal MaaS without anonymous GPU burn.

  • Leader Worker Set Operator: An orchestration primitive for leader/worker distributed pods, which is used by distributed inference patterns. It lets very large models span devices and nodes instead of forcing vertical-only designs.

  • cert-manager Operator: Provides issuers, certificates, and renewals for in-cluster TLS consumers, such as serving stacks. It eliminates manual cert rotation theater before inference goes live.

Architectural Reality Check for Distributed Inference
To deploy models by using Distributed Inference with the llm-d runtime, your cluster must be running OpenShift Container Platform version 4.20 or later. You must also have the cert-manager Operator, Red Hat Connectivity Link Operator, and Red Hat Leader Worker Set Operator installed.

3. Operators for Distributed Workloads & RAG (Heavy Machinery)

If your generative AI platform requires advanced workload queuing or Retrieval-Augmented Generation (RAG), you will also use:

  • Red Hat build of Kueue Operator: Controls quotas, priorities, and preemption policies for batch and serving workloads. It stops queue sprawl and aligns compute spend with business priority, not just the loudest team.

  • Llama Stack Operator: Data scientists and AI engineers can leverage Retrieval-Augmented Generation (RAG) capabilities provided by the integrated Llama Stack Operator. By combining large language model inference, semantic retrieval, and vector database storage, teams can obtain tailored, accurate, and verifiable answers to complex queries based on their own datasets.

  • Observability Operators: The Cluster Observability Operator, Tempo Operator, and Red Hat build of OpenTelemetry deploy an observability stack.
