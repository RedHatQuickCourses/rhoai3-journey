Lab 0 — Assembling the AI Engine Room

It is time to apply configuration as code.

In this lab, you are a platform engineer tasked with bootstrapping a greenfield OpenShift cluster into a Models-as-a-Service (MaaS) factory [cite: 4558-4559]. While you could install these operators one by one through the OpenShift Web Console UI, real factories are built with automation and code[cite: 4560].

Step 1: Gain CLI Access (The Web Terminal)

Before we can run our deployment scripts, we need a terminal connected to the cluster[cite: 4569].

  1. [cite_start]Log in to the OpenShift Web Console as a cluster administrator[cite: 4570].

  2. Navigate to Ecosystem → Software Catalog (or OperatorHub)[cite: 4571].

  3. Search for Web Terminal and click the Red Hat Web Terminal card[cite: 4572].

  4. Click Install and accept the default settings[cite: 4573].

  5. Wait for the installation to succeed, then refresh the page[cite: 4574].

  6. Look at the top right of the OpenShift Web Console header. You will now see a command-line icon (>_). [cite_start]Click it [cite: 4575-4576].

A terminal window will initialize at the bottom of your screen. Click start, and a pre-authenticated prompt will appear[cite: 4577]. Every command from here on is declarative.

Step 2: Lay the Hardware Foundation (NFD & GPU)

Now that you are in the terminal, we will use Kubernetes manifests (YAML) to instruct the Operator Lifecycle Manager (OLM) to subscribe to our required infrastructure operators[cite: 4579].

1. Node Feature Discovery (NFD) Operator

The NFD operator scans your nodes and applies specific hardware labels (e.g., highlighting that a node has an NVIDIA GPU). The GPU operators rely entirely on these labels to know where to install their drivers [cite: 4580-4581].

Create the Namespace: Always start with a fresh namespace to avoid OLM conflicts[cite: 4584].

oc create namespace openshift-nfd

Create the OperatorGroup: Use oc create (not oc apply) with generateName to allow OLM to dynamically generate a unique name, preventing state conflicts [cite: 4586-4587].

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: openshift-nfd-
  namespace: openshift-nfd
spec:
  targetNamespaces:
    - openshift-nfd
EOF

Create the Subscription: This subscribes the namespace to the Red Hat catalog, triggering the OLM to download and install the NFD Operator [cite: 4598-4599].

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: nfd
  namespace: openshift-nfd
spec:
  channel: "stable"
  installPlanApproval: Automatic
  name: nfd
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Create the Custom Resource: Installing the operator only deploys the controller. To deploy the daemonsets that actually scan your nodes for hardware features, create the Custom Resource [cite: 4617-4618].

cat <<EOF | oc create -f -
apiVersion: nfd.openshift.io/v1
kind: NodeFeatureDiscovery
metadata:
  name: nfd-instance
  namespace: openshift-nfd
spec: {}
EOF

2. NVIDIA GPU Operator

With NFD labeling your nodes, deploy the GPU operator to inject the drivers[cite: 4634].

Create a Dedicated Namespace: Always give the GPU operator its own namespace to prevent OLM tracking conflicts [cite: 4644-4645].

oc create namespace nvidia-gpu-operator

Create the OperatorGroup: [source,bash]

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: nvidia-gpu-operator-
  namespace: nvidia-gpu-operator
spec:
  targetNamespaces:
    - nvidia-gpu-operator
EOF

Dynamically Find the Channel and Subscribe: Operator channels change as new versions are released. Hardcoding the channel often breaks automated scripts. This block dynamically queries the catalog for the correct default channel before creating the Subscription [cite: 4660-4663].

CHANNEL=$(oc get packagemanifest gpu-operator-certified -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: gpu-operator-certified
  namespace: nvidia-gpu-operator
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Automatic
  name: gpu-operator-certified
  source: certified-operators
  sourceNamespace: openshift-marketplace
EOF

Deploying the ClusterPolicy: The operator controller is running, but it does nothing until a ClusterPolicy is defined[cite: 4688]. You must explicitly define the required schema blocks and configure the runtime for OpenShift (crio) [cite: 4691-4692].

cat <<EOF | oc create -f -
apiVersion: nvidia.com/v1
kind: ClusterPolicy
metadata:
  name: gpu-cluster-policy
spec:
  operator:
    defaultRuntime: crio
  driver:
    enabled: true
  toolkit:
    enabled: true
  devicePlugin:
    enabled: true
  dcgm:
    enabled: true
  dcgmExporter:
    enabled: true
  gfd:
    enabled: true
  nodeStatusExporter:
    enabled: true
  daemonsets: {}
EOF

Step 3: Deploy the Networking & Distributed Serving Layer

To serve models securely and across multiple nodes, we need our certificates and distributed workload managers [cite: 4723-4724].

1. cert-manager Operator

This operator is a prerequisite for RHOAI 3.3, as RHOAI relies heavily on automated TLS certificate generation for its internal communication and model-serving components[cite: 4726].

oc create namespace cert-manager-operator

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: openshift-cert-manager-
  namespace: cert-manager-operator
spec:
  targetNamespaces:
    - cert-manager-operator
EOF

CHANNEL=$(oc get packagemanifest openshift-cert-manager-operator -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: openshift-cert-manager-operator
  namespace: cert-manager-operator
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Automatic
  name: openshift-cert-manager-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF
Autonomous Initialization

Unlike other operators that require you to manually define a Custom Resource to start the actual workload, the cert-manager Operator is designed to be autonomous out of the box [cite: 4758-4759]. It will automatically spin up the certificate-issuing pods in a secondary namespace called cert-manager[cite: 4760].

2. LeaderWorkerSet (LWS) Operator

LWS is a critical dependency for distributed workloads, allowing large AI training and inference tasks to be split across multiple nodes and GPUs using a coordinated leader-worker pod topology [cite: 4771-4772].

oc create namespace openshift-lws-operator

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: openshift-lws-operator-
  namespace: openshift-lws-operator
spec:
  targetNamespaces:
    - openshift-lws-operator
EOF

CHANNEL=$(oc get packagemanifest lws-operator -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: lws-operator
  namespace: openshift-lws-operator
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Automatic
  name: lws-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Create the Custom Resource: Red Hat expects this specific Custom Resource to be named cluster [cite: 4815-4816].

cat <<EOF | oc create -f -
apiVersion: operator.openshift.io/v1
kind: LeaderWorkerSetOperator
metadata:
  name: cluster
  namespace: openshift-lws-operator
spec:
  managementState: Managed
  logLevel: Normal
  operatorLogLevel: Normal
EOF

apiVersion: operators.coreos.com/v1alpha1

Step 4: Workload Coordination (Kueue)

Kueue manages the actual job queues, quotas, and resource sharing. It holds jobs in a queue and only releases them to the cluster when the requested hardware is actually available [cite: 4896-4897].

Cluster-Wide Mode Tactic: Because Kueue needs to watch for workloads submitted by data scientists across any project in the cluster, we must configure the OperatorGroup to use Cluster-wide tracking (by omitting the targetNamespaces block) [cite: 4898, 4907-4908].

oc create namespace openshift-kueue-operator

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: openshift-kueue-operator-
  namespace: openshift-kueue-operator
EOF

CHANNEL=$(oc get packagemanifest kueue-operator -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: kueue-operator
  namespace: openshift-kueue-operator
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Automatic
  name: kueue-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Create the Kueue Custom Resource: [source,bash]

cat <<EOF | oc create -f -
apiVersion: kueue.openshift.io/v1
kind: Kueue
metadata:
  name: cluster
  namespace: openshift-kueue-operator
spec: {}
EOF

Step 5: Install the Meta-Orchestrator (OpenShift AI)

This is the final step in the prerequisites phase for building a Models-as-a-Service (MaaS) solution[cite: 4958].

Management Tactic: Because we are building a highly specific MaaS stack, we want to pin the installation to the stable-3.3 channel to ensure we get the correct operator version rather than accidentally floating to a future major release that might introduce breaking changes to our MaaS architecture [cite: 4983-4984].

oc create namespace redhat-ods-operator

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: redhat-ods-operator-
  namespace: redhat-ods-operator
EOF

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: rhods-operator
  namespace: redhat-ods-operator
spec:
  channel: "stable-3.3"
  installPlanApproval: Automatic
  name: rhods-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

This guide intentionally stops after installing the Operator. It does not deploy the DataScienceCluster (DSC) or DSCInitialization (DSCI) Custom Resources [cite: 4959-4960]. Configuring the actual data science environment will be handled in the next section of this enablement journey[cite: 4961].
