Strategy: Defining Your Path to Production (OpenShift AI 3.3)

Before we write a single line of YAML or commit to a Git repository, we have to help the customer choose their architectural strategy.

In Red Hat OpenShift AI 3.3, "model serving" has split into two very specific, mechanically different paths. Choosing the wrong path leads to either wasted infrastructure spend or an inability to apply governance.

Our job is to guide the customer. We need to look at their use cases and explain why not every AI workload should be deployed the same way.

The Strategic Divide: Standalone vs. Distributed MaaS

Not every AI workload is the same. Serving a massive, fine-tuned model for a single, high-frequency internal application requires a different architecture than providing a general-purpose, rate-limited LLM API to 500 enterprise developers across different departments.

When positioning OpenShift AI to leadership, think of it like transportation:

  • Standalone Serving is a Race Car. It is purpose-built, highly tuned for a single driver (application), and consumes dedicated resources. It’s incredibly fast, but it doesn’t scale to handle the general public.

  • Models as a Service (MaaS) is City Transit. It is designed for high throughput, shared resources, and access control (tickets/tokens). It efficiently moves hundreds of people (developers) using the same infrastructure footprint.

Path A: Standalone Serving (The "Race Car")

  • The Setup: A traditional KServe deployment where a model gets a dedicated OpenShift Route and dedicated GPU(s).

  • The Pros: Absolute performance isolation. Zero interference from other workloads.

  • The Cons: Terrible resource utilization if the application isn’t constantly inferencing. There is no built-in rate limiting, no API key generation in the UI, and no multi-tenant token tracking.

  • When to pitch it: When a customer has a single, mission-critical application that demands guaranteed, uninterrupted latency and the business unit is willing to pay for dedicated hardware.

Path B: Distributed MaaS (The "City Transit")

  • The Setup: The model is deployed using the Distributed Inference Server (llm-d), published as a MaaS endpoint, and placed behind the Kubernetes Gateway API and Kuadrant governance engine.

  • The Pros: Massive ROI. GPUs are pooled and shared. Developers are issued API keys and placed into Service Tiers (e.g., Free, Premium) to prevent any one team from exhausting the system. Cross-departmental chargebacks become possible.

  • The Cons: Requires a much more complex architectural foundation (which we will cover next).

  • When to pitch it: When the business wants to become an "internal AI provider" for its developers, offering a standard catalog of approved models (like Granite or Llama) to the entire organization securely.

Next Steps: Understanding the Blueprint

Now that we know why we are pitching MaaS to maximize the customer’s hardware ROI, we need to understand how it actually works.

If we just click "Deploy" in the UI without understanding the backend, we will fail. In the next module, we will look under the hood at the "Invisible Dependency Chain" of operators that make this City Transit system run.
