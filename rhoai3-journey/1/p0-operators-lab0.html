Lab: Assembling the AI Engine Room

It is time to apply configuration as code.

In this lab, you are a platform engineer tasked with bootstrapping a greenfield OpenShift cluster into a Models-as-a-Service (MaaS) factory. While you could install these operators one by one through the OpenShift Web Console UI, real factories are built with automation and code.

To execute this, we will first install the Red Hat Web Terminal Operator via the UI. This provides a secure, in-cluster command-line interface directly within your browser. From there, we will deploy the rest of the MaaS operator stack using the oc CLI.

Prerequisites

Access to an OpenShift cluster with cluster-admin privileges.

Step 1: Gain CLI Access (The Web Terminal)

The Web Terminal Operator installs the DevWorkspace Operator as a dependency.

Before we can run our deployment scripts, we need a terminal connected to the cluster.

  1. Log in to the OpenShift Web Console as a cluster administrator.

  2. Navigate to Ecosystem → Software Catalog (or OperatorHub).

  3. Search for Web Terminal and click the Red Hat Web Terminal card.

  4. Click Install and accept the default settings.

  5. Wait for the installation to succeed, refresh the page.

  6. Launch the Terminal: Look at the top right of the OpenShift Web Console header. You will now see a command-line icon (>_). Click it.

A terminal window will initialize at the bottom of your screen. Click start, and a pre-authenticated prompt will appear. After the web terminal is initialized, you can use the preinstalled CLI tools like oc, kubectl, odo, kn, tkn, helm, and subctl in the web terminal.

Step 2: Lay the Hardware Foundation (NFD & GPU)

Now that you are in the terminal, we will use Kubernetes manifests (YAML) to instruct the Operator Lifecycle Manager (OLM) to subscribe to our required infrastructure operators.

The NFD operator scans your nodes and applies specific hardware labels (e.g., highlighting that a node has an NVIDIA H100 or an AMD MI300X). The GPU operators rely entirely on these labels to know where to install their drivers.

Procedure: Install the NFD Operator

Create the Namespace

Always start with a fresh namespace to avoid Operator Lifecycle Manager (OLM) conflicts.

oc create namespace openshift-nfd

Create the OperatorGroup

Use oc create (not oc apply) to allow OLM to dynamically generate a unique name for the OperatorGroup, preventing state conflicts.

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: openshift-nfd-
  namespace: openshift-nfd
spec:
  targetNamespaces:
  - openshift-nfd
EOF

Create the Subscription

This subscribes the namespace to the Red Hat catalog, triggering the OLM to download and install the NFD Operator.

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: nfd
  namespace: openshift-nfd
spec:
  channel: "stable"
  installPlanApproval: Automatic
  name: nfd
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Verify Operator Installation

Wait a few moments, then check the ClusterServiceVersion (CSV) to ensure the operator installed successfully.

oc get csv -n openshift-nfd

Note: Proceed to next step only when the PHASE displays as Succeeded.

Create the NodeFeatureDiscovery Instance

Installing the operator only deploys the controller. To deploy the daemonsets that actually scan your nodes for hardware features, create the Custom Resource.

cat <<EOF | oc create -f -
apiVersion: nfd.openshift.io/v1
kind: NodeFeatureDiscovery
metadata:
  name: nfd-instance
  namespace: openshift-nfd
spec: {}
EOF

Verify Hardware Discovery

Check that the NFD master and worker pods are running:

oc get pods -n openshift-nfd

To verify that NFD is successfully labeling your nodes with hardware features, pick a worker node and describe it:

oc describe node <your-worker-node-name> | grep feature.node.kubernetes.io

The GenAI Value: NFD is the foundation of hardware governance. By labeling your nodes, NFD allows you to subsequently deploy the NVIDIA or AMD operators and configure precise Hardware Profiles. This prevents developers from accidentally scheduling an expensive H100 GPU for a simple CPU-bound data preprocessing task.

Deploy the NVIDIA GPU Operator:

With NFD labeling your nodes, deploy the GPU operator to inject the drivers.

You can install this operator without GPU nodes present. The driver daemonsets will simply remain unscheduled until a GPU node is added to the cluster._

Installing the Operator

Create a Dedicated Namespace

Always give the GPU operator its own namespace to prevent OLM tracking conflicts.

oc create namespace nvidia-gpu-operator

Create the OperatorGroup

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: nvidia-gpu-operator-
  namespace: nvidia-gpu-operator
spec:
  targetNamespaces:
  - nvidia-gpu-operator
EOF

Dynamically Find the Channel and Subscribe

Operator channels change as new versions are released (e.g., v24.09 to v25.10). Hardcoding the channel often breaks automated scripts. This block dynamically queries the catalog for the correct default channel before creating the Subscription. 
# Fetch the current default channel
CHANNEL=$(oc get packagemanifest gpu-operator-certified -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

# Create the subscription using the discovered channel
cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: gpu-operator-certified
  namespace: nvidia-gpu-operator
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Automatic
  name: gpu-operator-certified
  source: certified-operators
  sourceNamespace: openshift-marketplace
EOF

OLM Troubleshooting Tactics

If the operator does not immediately install, OLM might be stuck. Use these commands to diagnose the silent failures:

  • Check if OLM registered the Subscription:

     oc get subscription gpu-operator-certified -n nvidia-gpu-operator -o wide

  • Check if an InstallPlan was generated or failed:

    oc get installplan -n nvidia-gpu-operator

  • Verify the ClusterServiceVersion (CSV) succeeded:

    oc get csv -n nvidia-gpu-operator
Do not proceed until the CSV phase shows as Succeeded.

Deploying the ClusterPolicy

The operator controller is running, but it does nothing until a ClusterPolicy is defined.

In older versions, you could pass an empty spec: {} and a mutating webhook would fill in the blanks. In versions v25.10 and later, the Custom Resource Definition enforces strict OpenAPI validation. You must explicitly define the required schema blocks and configure the runtime for OpenShift (crio). 
cat <<EOF | oc create -f -
apiVersion: nvidia.com/v1
kind: ClusterPolicy
metadata:
  name: gpu-cluster-policy
spec:
  operator:
    defaultRuntime: crio
  driver:
    enabled: true
  toolkit:
    enabled: true
  devicePlugin:
    enabled: true
  dcgm:
    enabled: true
  dcgmExporter:
    enabled: true
  gfd:
    enabled: true
  nodeStatusExporter:
    enabled: true
  daemonsets: {}
EOF

Verification

Once the ClusterPolicy is accepted, the operator will attempt to roll out the NVIDIA driver daemonsets to any NFD-labeled GPU nodes.

Watch the operator build the environment:

oc get clusterpolicy gpu-cluster-policy
oc get pods -n nvidia-gpu-operator

Note: If you have no GPU nodes, the pods responsible for the device plugin and drivers will not deploy, which is expected behavior.

Step 3: Deploy the Networking & Distributed Serving Layer

To serve models securely and across multiple nodes, we need our certificates and distributed workload managers. RHOAI relies heavily on automated TLS certificate generation for its internal communication, webhooks, and model-serving components.

Installing the Cert-manager Operator

Create the Dedicated Namespace

Red Hat’s recommended namespace for the operator controller is cert-manager-operator.

oc create namespace cert-manager-operator

Create the OperatorGroup

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: openshift-cert-manager-
  namespace: cert-manager-operator
spec:
  targetNamespaces:
  - cert-manager-operator
EOF

Dynamically Find the Channel and Subscribe

# Fetch the current default channel
CHANNEL=$(oc get packagemanifest openshift-cert-manager-operator -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

# Create the subscription using the discovered channel
cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: openshift-cert-manager-operator
  namespace: cert-manager-operator
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Automatic
  name: openshift-cert-manager-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Verification

Unlike other operators that require you to manually define a Custom Resource to start the actual workload, the cert-manager Operator for Red Hat OpenShift is designed to be autonomous out of the box.

Once the operator successfully installs, it will automatically generate a CertManager custom resource named cluster and spin up the actual certificate-issuing pods in a brand new, secondary namespace called cert-manager.

Verify the Operator Installed

Wait a few moments, then check the ClusterServiceVersion (CSV) to ensure the operator downloaded successfully.

oc get csv -n cert-manager-operator

Note: Proceed to the next step only when the PHASE displays as Succeeded.

Verify the Operands Deployed Automatically

Check the automatically created cert-manager namespace to ensure the controller, cainjector, and webhook pods are all initialized and running:

oc get pods -n cert-manager

Once those pods are in a Running state, your cluster is fully capable of provisioning and rotating its own internal TLS certificates. You are now ready to proceed with the Red Hat OpenShift AI 3.3 installation!

Create the LeaderWorkerSetOperator Custom Resource

The operator is installed, but it needs a configuration file to deploy the actual LWS components. Red Hat expects this specific Custom Resource to be named cluster.

apiVersion: operator.openshift.io/v1

  operatorLogLevel: Normal

Verification

Verify that the LWS controller manager pods are running:

Once these pods show a status of Running, the LeaderWorkerSet APIs are successfully registered on your cluster and ready to be used by Red Hat OpenShift AI.

Installing the Red Hat Connectivity Link Operator

This guide walks through deploying the Red Hat Connectivity Link Operator. In OpenShift AI 3.3, this operator manages Kuadrant, which is strictly required if you want to expose Large Language Models (LLMs) securely using the Models-as-a-Service (MaaS) interface, enforce rate limits, and provide robust API governance.

Installing the Operator

Switch to the Global Operators Namespace

Red Hat’s default namespace for cluster-wide operators is openshift-operators.

oc project openshift-operators

[IMPORTANT]

Do not create an OperatorGroup. Because openshift-operators is a default system namespace, OpenShift has already provisioned a global OperatorGroup for it. Creating a second one will cause the Operator Lifecycle Manager (OLM) to freeze the installation.

Dynamically Find the Channel and Subscribe

Management Tactic: We query the redhat-operators catalog for the current default channel of the rhcl-operator package to ensure the installation remains resilient across OpenShift major releases.

# Fetch the current default channel
CHANNEL=$(oc get packagemanifest rhcl-operator -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

# Create the subscription using the discovered channel
cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: rhcl-operator
  namespace: openshift-operators
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Manual
  name: rhcl-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
  startingCSV: rhcl-operator.v1.3.1
EOF

Auto-Approve the Installation and Dependencies

Because we set the subscription to `Manual` to freeze the version, we must approve the initial `InstallPlan`. The Connectivity Link Operator also pulls in dependencies like DNS and Limitador, which will generate their own InstallPlans. This script automatically finds and approves all of them so you don't have to use the Web Console UI.
echo "Waiting for OLM to generate InstallPlans..."
sleep 15

# Find and approve any pending InstallPlans for our operators
for ip in $(oc get installplan -n openshift-operators -o jsonpath='{.items[?(@.spec.approved==false)].metadata.name}'); do
  echo "Approving InstallPlan: $ip"
  oc patch installplan $ip -n openshift-operators --type merge -p '{"spec":{"approved":true}}'
done

Deploying the Kuadrant Instance

Verify the Operator Controller Installed

Wait a few moments, then check the ClusterServiceVersion (CSV) to ensure the operator downloaded successfully.

oc get csv -n openshift-operators | grep rhcl

Note: Proceed to the next step only when the PHASE displays as Succeeded. If no response run the above Auto-Approve script again.

Create the Kuadrant Custom Resource

The operator is installed, but the system isn’t active until we deploy the Kuadrant control plane. Passing an empty spec: {} tells the operator to deploy the most optimal default configuration.

cat <<EOF | oc create -f -
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata:
  name: kuadrant
  namespace: openshift-operators
spec: {}
EOF

Verification

Once the Custom Resource is submitted, the operator will spin up the Kuadrant control plane pods, including the authorino (authentication) and limitador (rate-limiting) components.

Watch the pods spin up and ensure they transition to a Running state:

oc get pods -n openshift-operators -w

You can also explicitly verify that the Kuadrant system successfully initialized by checking its deployment conditions:

oc wait kuadrant/kuadrant --for="condition=Ready=true" -n openshift-operators --timeout=300s

Installing the Red Hat build of Kueue Operator

While LeaderWorkerSet handles the physical coordination of multi-node pods, Kueue is the "brain" that manages the actual job queues, quotas, and resource sharing. When multiple data science teams are submitting heavy AI workloads, Kueue ensures they don’t fight over the same GPUs. It holds jobs in a queue and only releases them to the cluster when the requested hardware (like our NFD-discovered GPUs) is actually available.

Here is an important detail for this specific walkthrough: Because Kueue needs to watch for workloads submitted by data scientists across any project in the cluster, we must configure the OperatorGroup to use Cluster-wide tracking (by omitting the targetNamespaces block).

Installing the Operator

Create the Dedicated Namespace

Red Hat’s official tracking namespace for the Kueue controller is openshift-kueue-operator.

oc create namespace openshift-kueue-operator

Create the OperatorGroup (Cluster-Wide Mode)

Management Tactic: Kueue must intercept and manage job queues across the entire cluster. By using oc create and intentionally omitting the targetNamespaces block from the OperatorGroup, we force Operator Lifecycle Manager (OLM) to deploy the operator in cluster-wide (AllNamespaces) mode.

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  generateName: openshift-kueue-operator-
  namespace: openshift-kueue-operator
EOF

Dynamically Find the Channel and Subscribe

Management Tactic: We query the redhat-operators catalog for the current default channel of the kueue-operator package to ensure the install script dynamically adapts to future OpenShift versions.

# Fetch the current default channel
CHANNEL=$(oc get packagemanifest kueue-operator -n openshift-marketplace -o jsonpath='{.status.defaultChannel}')

# Create the subscription using the discovered channel
cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: kueue-operator
  namespace: openshift-kueue-operator
spec:
  channel: "${CHANNEL}"
  installPlanApproval: Automatic
  name: kueue-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Deploying the Kueue Instance

Verify the Operator Controller Installed

Wait a few moments, then check the ClusterServiceVersion (CSV) to ensure the operator downloaded and initialized successfully.

oc get csv -n openshift-kueue-operator

Note: Proceed to the next step only when the PHASE displays as Succeeded.

Create the Kueue Custom Resource

The operator is installed, but the queuing engine isn’t active until we deploy the central Kueue Custom Resource. Red Hat expects this configuration to be named cluster. Passing an empty spec: {} tells the webhook to inject the optimal default configurations.

cat <<EOF | oc create -f -
apiVersion: kueue.openshift.io/v1
kind: Kueue
metadata:
  name: cluster
  labels:
    app.kubernetes.io/managed-by: kustomize
    app.kubernetes.io/name: kueue-operator
spec:
  config:
    integrations:
      frameworks:
        - BatchJob
  logLevel: Normal
  operatorLogLevel: Normal
  managementState: Managed
EOF

Verification

Once the Custom Resource is submitted, the operator will spin up the Kueue controller manager pods, which actively watch for ClusterQueue and LocalQueue objects.

Verify that the Kueue controller manager pods are running:

oc get pods -n openshift-kueue-operator

Once these pods show a status of Running, Kueue is successfully registered on your cluster. Data science teams can now submit AI training tasks, and Kueue will hold them in line until the cluster has enough resources to execute them.

Install OpenShift AI Operator

This section walks through deploying the core Red Hat OpenShift AI Operator. This is the final step in the prerequisites phase for building a Models-as-a-Service (MaaS) solution.

This guide intentionally stops after installing the Operator. It does not deploy the DataScienceCluster (DSC) or DSCInitialization (DSCI) Custom Resources. Configuring the actual data science environment will be handled in the next section of this enablement journey.

Create the Dedicated Namespace

oc create namespace redhat-ods-operator

apiVersion: operators.coreos.com/v1
kind: OperatorGroup

  generateName: redhat-ods-operator-
  namespace: redhat-ods-operator

Subscribe to the stable-3.3 Channel

cat <<EOF | oc create -f -
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: rhods-operator
  namespace: redhat-ods-operator
spec:
  channel: "stable-3.3"
  installPlanApproval: Automatic
  name: rhods-operator
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF

Verification

Verify the Operator Controller Installed

Wait a few moments for OLM to process the subscription, then check the ClusterServiceVersion (CSV) to ensure RHOAI downloaded successfully.

oc get csv -n redhat-ods-operator

You should see output similar to this, confirming version 3.3.x is active: rhods-operator.3.3.0 Red Hat OpenShift AI 3.3.0 …​ Succeeded

Verify the Controller Pods are Running

Check that the operator pod has successfully spun up in the namespace:

oc get pods -n redhat-ods-operator

Once the rhods-operator pod shows a status of Running, the OpenShift AI APIs are successfully registered on your cluster.
