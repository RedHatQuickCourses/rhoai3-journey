Lab: Customizing the Catalog & Supply Chain

Your organization’s custom models deserve the same seamless deployment experience as third-party foundation models.

In OpenShift AI 3.3, administrators have the option to manage catalog sources directly through the new Dashboard UI. However, for true enterprise scale, Platform Engineers must understand how to manage these sources declaratively. This allows for strict "GitOps" workflows where every model approved for deployment is tracked as code.

As an OpenShift cluster administrator, you configure a custom model catalog source by adding an entry to the model-catalog-sources config map.

In this lab, you will use the "Embedded Strategy". This is the most robust method for air-gapped or strict firewall environments. You will configure the ConfigMap to serve two roles simultaneously:

  1. The Pointer (sources.yaml): It defines the catalog name, ID, and where to look for the model definitions.

  2. The Database (e.g., acme-models.yaml): It contains the actual metadata and external storage URIs for your proprietary models.

Procedure: The Declarative GitOps Strategy

1. Target the Registry Namespace

First, ensure you are working in the correct project where the Model Registry and Catalog components are running.

oc project rhoai-model-registries

2. Open the Catalog Configuration

The Model Catalog uses a Kubernetes ConfigMap to define its sources. You will edit this object directly.

oc edit configmap model-catalog-sources

You are editing a ConfigMap, not a local file. The sources.yaml configuration and your custom model catalog file will exist as sibling entries inside the data section of this object.

3. Configure the Source and Add Models

You need to define your custom source under the catalogs field in sources.yaml, and provide the actual model payload in a sibling file.

Carefully add the following structure into the data: section of the ConfigMap:

data:
  # KEY 1: The Registration (The "Pointer")
  sources.yaml: |-
    catalogs:
    - name: Acme Corp Private    # The user-friendly name of the catalog source.
      id: acme_custom_catalog    # The unique ID of the catalog source.
      type: yaml                 # The catalog source type.
      enabled: true              # Whether the catalog source is enabled.
      properties:
        yamlCatalogPath: acme-models.yaml # The location of your catalog source file.
      labels:
        - Acme AI Lab            # Optional labels for your catalog source to display in the catalog.

  # KEY 2: The Inventory (The "Database")
  # This key contains the actual Model definitions and external storage URIs.
  acme-models.yaml: |-
    models:
      - name: "Acme-Finance-7B"
        provider: "Acme AI Lab"
        description: "Internal fine-tuned model stored in secure S3."
        customProperties:
          model_type:
            metadataType: MetadataStringValue
            string_value: "generative"
          author:
            metadataType: MetadataStringValue
            string_value: "Acme Data Science"
        artifacts:
          - name: "model-artifact"
            uri: "s3://acme-private-models/Finance-7B-v1.2"
YAML Indentation Warning

Notice the structure above. The acme-models.yaml content is added in the same config map as a sibling of sources.yaml. Ensure the indentation of the inner YAML is preserved exactly as shown, or the catalog parser will fail.

Step 4: Verification and UI Categorization

These changes might take a few minutes to be displayed in the catalog on the dashboard.

  1. Open the OpenShift AI dashboard and click AI hub → Catalog.

  2. Look at the category menu bar at the top of the catalog:

    • If you used the labels field (as we did with "Acme AI Lab"): Your models will appear in a custom category named <your-label-name> models.

    • If you omitted the labels field: Your custom models will be aggregated under the Other models category.

  3. Click your custom model to view its details page. You will see that the deployment wizard is now ready to pull your custom asset directly from the external S3 storage URI you provided.

You have successfully engineered the supply chain to serve private assets declaratively. Next, we will cover the Day 2 Operations required to maintain, secure, and govern this system.
