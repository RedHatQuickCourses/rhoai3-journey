Industrializing AI: The Business Value of Models as a Service (OpenShift AI 3.3)

Technology Preview Notice

Models-as-a-Service (MaaS) is currently available in Red Hat OpenShift AI 3.3 as a Technology Preview feature. While we are actively building out these capabilities to drive business value, they are not yet supported with production service level agreements (SLAs). This course is designed to equip you with a GitOps framework to build proofs-of-concept and understand the architecture ahead of General Availability (GA).

If you’ve followed the rest of this OpenShift AI series, you already know how to serve a model. But here’s the reality check from the field: simply hosting a model isn’t enough anymore.

Right now, businesses are struggling with the polarizing forces of Generative AI: development teams demand rapid access to LLMs, but IT requires rigorous governance and cost control.

The old "ticket-based" workflow—where developers request dedicated GPUs for every isolated experiment—is a severe operational bottleneck. It leads to idle hardware, massive cloud bills, and "Shadow AI" where developers bypass IT to use public APIs, leaking proprietary enterprise data in the process.

With the release of Red Hat OpenShift AI 3.3, we have a strategic answer: Models as a Service (MaaS). We can now stop treating AI like a series of isolated science projects and start running it like an industrialized factory.

The Core Value Proposition: The Token Factory

When we architect this solution, our goal is to help the business shift from consuming public AI to becoming an internal AI provider. We need to surface three specific business returns:

  1. Sovereign AI & Data Privacy: Use cases that cannot run on public services due to privacy and regulatory concerns can now be served locally. By hosting validated open-source models centrally on OpenShift, no proprietary data ever leaves the hybrid environment.

  2. Accelerated Enterprise Adoption: By removing the infrastructure bottleneck, developers no longer wait weeks for GPU provisioning. They simply log into the Gen AI Studio, request an API key, and instantly integrate governed LLM endpoints into their applications.

  3. Maximizing ROI (The Token Factory): Dedicated hardware per project destroys ROI. By pooling resources using Distributed Inference (llm-d), rate-limiting, and quota management, we build a highly utilized "token generation factory at scale." The business serves the entire organization from a consolidated, cost-efficient hardware footprint.

The Operational Divide: Who Operates the Factory?

To successfully deliver this at scale, we have to bridge the gap between two distinct personas. This course will walk you through how to build for both sides of the house:

  • The Platform Engineer (Infrastructure): They care about the ecosystem. We’ll dive into how to configure the Kubernetes Gateway API, ensure the governance engine (Kuadrant) is running, and allocate global GPU quotas.

  • The AI Administrator (Business Logic): They care about chargebacks and governance. We’ll explore how to use the Gen AI Studio to define Service Tiers, assign user groups, deploy models, and monitor token consumption.

Your Learning Journey

Because this is the capstone of our RHOAI series, we are taking a GitOps-First approach. While OpenShift AI 3.3 introduces a beautiful UI, true enterprise scale requires Infrastructure-as-Code. We learned some hard lessons building this stack ourselves, so we are going to teach you how the ecosystem actually works beneath the UI.

Here is how we will break down the journey:

Module Your Enablement Goal

1. The Business Case & Strategy

The Pitch: Learn when to use Standalone serving versus Distributed MaaS (llm-d) to maximize hardware ROI and fit the right architectural pattern.

2. Architecture & Ecosystem (Lessons from the Trenches)

The Blueprint: Understand the "Invisible Dependency Chain." Discover how the MaaS stack relies on the Gateway API for routing, Kuadrant for authentication, Kueue for resource quotas, and llm-d for caching.

3. Building the Foundation (The Platform Engineer Track)

The Build: A strict GitOps/CLI lab. You will deploy the prerequisites via YAML, enable the MaaS backend, configure the Gateway, and apply RBAC roles for tier-based governance.

4. Governing Models via UI (The AI Admin Track)

The Demo: Master the Gen AI Studio dashboard. You will learn how to modify default Free/Premium Service Tiers, map user groups, deploy models visually, and generate API keys.

5. Day-2 Operations & Troubleshooting

The Support: Equip the operations team to maintain the factory. Learn to diagnose Gateway authorization errors (401/403), quota enforcement blocks (429), and resolve pending pod issues.
