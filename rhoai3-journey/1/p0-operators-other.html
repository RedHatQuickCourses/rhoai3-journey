Summary, handoff & troubleshooting

The foundation is poured. The machinery is wired. Now organize the supply chain—data, models, profiles—not more operators.

Infrastructure checkpoint

Before leaving Phase 0, confirm:

  • Web Terminal — browser oc works for GitOps execution.

  • Red Hat OpenShift AI operator CSV Succeeded in redhat-ods-operator.

  • NFD workers running; nodes expose accelerator labels after GPU operator install (GPU operator is usually a separate subscription—follow Working with accelerators in docs).

  • cert-manager CSV Succeeded in cert-manager-operator.

  • Connectivity Link CSV Succeeded in openshift-operators.

  • DataScienceCluster status Ready; dashboard route resolves.

  • Leader Worker Set CSV Succeeded (if you completed Lab 4).

Handoff — what to run next

You are ready for Phase 1 — Assembling the factory in the broader journey:

  1. Storage / data connections — bind object storage without credential sprawl.

  2. Model registry — connect backing database; govern artifact promotion.

  3. Model catalog — curate what developers may deploy.

  4. Hardware profiles & Kueue — turn labeled GPUs into shared capacity with queues.

Keep using Git for Subscriptions and for DataScienceCluster revisions—treat the console as a read-only dashboard, not the source of truth.

Troubleshooting — operator and control-plane class failures

Symptoms below are deliberately practical; always cross-check with the Installing and Managing OpenShift AI PDFs in rhoai33docs.

CSV stuck in Pending or Installing

  • Cause: Image pull, missing OperatorGroup, or manual InstallPlan not approved.

  • Checks: oc get installplan -n <operator-ns>, oc describe csv -n <operator-ns> <csv-name>.

  • Fix pattern: Approve InstallPlan, fix pull secrets for disconnected clusters, verify CatalogSource health (oc get catalogsource -n openshift-marketplace).

Subscription reports ConstraintsNotSatisfiable

  • Cause: Channel/catalog mismatch or conflicting installed operators.

  • Fix: Align channel with documentation for your OpenShift + RHOAI combination; remove obsolete CatalogSources if your platform team rotated them.

DataScienceCluster never reaches Ready

  • Cause: Unsatisfied component dependencies (for example serving prerequisites), storage class issues for PVCs, or invalid managementState combinations.

  • Checks: oc describe datasciencecluster default-dsc, operator logs in redhat-ods-operator, and component namespaces for CrashLoop pods.

  • Fix: Fix the first failing component in status—downstream errors are often noise.

Dashboard / gateway routes — HTTP 404 or TLS errors

  • 404: Route exists but backing service not ready—finish CSV/DSC reconciliation first.

  • TLS: cert-manager not installed or issuer not configured for the serving stack—revisit Lab 2.

  • Corporate MITM proxies: Clients may need trust anchors; that is an enterprise PKI problem, not an AI model problem.

Kuadrant / gateway pods missing

  • Cause: Connectivity Link not finished installing, or namespace changed between versions.

  • Fix: Verify CSV, then search cluster-wide (oc get pods -A | grep -i kuadrant). Consult Govern LLM access with Models-as-a-Service for the exact namespace layout for your release.

When in doubt, reduce the problem: uninstall the last operator added, confirm cluster stability, reinstall on the documented channel, then reapply DataScienceCluster only after all prerequisites are green.
