Capabilities & Features

To build a factory that scales, you must master the machinery behind the scenes.

As a Platform Engineer, you are not just managing disks; you are managing the lifecycle of data as it transitions from "Cold Object Storage" to "Hot Shared Memory".

OpenShift AI 3.3 introduces a suite of capabilities designed to abstract complex storage physics away from the data scientist, while giving you, the architect, absolute control over performance and security.

Here is the menu of capabilities that form your AI Data Supply Chain.

1. Data Connections

To achieve Zero-Touch Security and environment parity, RHOAI 3 separates the definition of a credential from the usage of a credential. We use a "Socket and Plug" model:

  • Connection Types (The Socket): Think of this as the "Electrical Socket" on the wall. The Administrator defines the shape of the socket to ensure safety and standardization. This tells the dashboard how to render a form and how to map user inputs to standard Environment Variables. While default types exist for S3 and OCI, you can create custom types for structured data like Snowflake, MongoDB, MySQL, or Milvus.

  • Data Connections (The Plug): This is the specific instance where a user provides their credentials to fit the socket. It is stored securely as a Kubernetes Secret and injected into Pods at runtime.

The "Day 1" Secret: Visibility vs. Routing

Many Platform Engineers deploy connection secrets via GitOps that route perfectly but fail to appear in the Data Science dashboard. You must explicitly add these two labels:

  • Frontend Visibility: You must apply the label opendatahub.io/managed: "true". This tells the OpenShift AI Dashboard to expose this connection to Data Scientists in the UI. Without it, the connection is functionally invisible.

  • Backend Routing: You must apply the annotation opendatahub.io/connection-type-protocol: <type> (e.g., s3, oci, uri). This tells the underlying Inference Service or Pipeline engine how to mechanically pull the data.

2. The Multi-Protocol Engine

Not all data moves the same way. RHOAI provides multiple native protocols to optimize for capacity, speed, or precision:

Protocol The Role The Physics & Usage

s3://

The Warehouse

Standard S3-compatible object storage used for massive datasets, data lakes, and standard model serving.

oci://

The Accelerator

High-speed model inference and distribution via "Modelcars". Instead of downloading loose files, models are packaged as container images. It leverages the node’s CRI-O container cache. If 10 users need the same model, the node pulls the layers once, and all 10 pods start instantly, producing massive deduplication.

pvc://

The Air-Gapped Drive

Allows deployments to route directly to a local Persistent Volume Claim (PVC), which is critical for highly secure, disconnected environments.

uri:// / hf://

The Precision Link

Provides direct HTTP/HTTPS access to specific model versions or files without exposing the entire root bucket.

3. Cluster Storage: The Physics of Workspaces (RWO vs. RWX)

Once data arrives from the "Warehouse," it needs a high-speed place to live while it is being processed. This is governed by your OpenShift Storage Classes. You must align the storage physics to your developers' collaboration needs:

  • ReadWriteOnce (RWO) - "The Scratchpad": High-speed block storage (e.g., AWS EBS, vSphere). It maps 1:1 with a user’s Workbench. Because it is physically incapable of attaching to multiple nodes simultaneously, it provides strict private storage.

  • ReadWriteMany (RWX) - "The Collaboration Zone": Network file storage (e.g., CephFS, NFS). It mounts to multiple nodes simultaneously. It is strictly required for shared team folders, sharing models across developers, and distributed training architectures.

4. Dual-Storage Pipeline Workspaces

In previous versions, AI Pipelines passed intermediate data between tasks by constantly uploading to and downloading from an S3 bucket (with a charge for each read / write). OpenShift AI 3.3 fundamentally shifts this by adopting a Dual-Storage (Hybrid) Architecture:

  • The Archive (S3): An S3-compatible connection is strictly required to instantiate the pipeline server. It acts as the permanent control-plane archive, storing the compiled pipeline YAML, step-by-step execution logs, and final output artifacts.

  • The Workspace (PVC): To eliminate cloud egress costs and latency, RHOAI 3.3 utilizes Persistent Volume Claims (PVCs) for the data plane. The pipeline engine dynamically creates an ephemeral shared PVC during the pipeline run. Components exchange massive intermediate datasets directly via the local cluster disk.
