Day 2 Operations & Enterprise Governance

A catalog that doesn’t load delivers zero value. A catalog that is governed delivers a competitive advantage.

You have built the supply chain. You have learned how to inject custom sources declaratively and deploy with hardware-aligned profiles. Now, we turn to Day 2 Operations: enforcing corporate governance, handling upgrades, and articulating the value of what you have built.

1. Curation and Compliance (The Allow/Disallow Engine)

One of the most powerful Day 2 capabilities introduced in OpenShift AI 3.3 is the ability to actively govern the showroom. Curation is not a one-time event; it is an ongoing operational requirement.

If your security team flags a specific model license, or if corporate policy dictates that only internally fine-tuned models can be used for a specific period, you do not need to tear down your infrastructure. The model catalog now provides an administrative capability in the OpenShift AI dashboard to selectively hide, disallow list, or remove specific models from the visible catalog.

This ensures rapid compliance with internal security, policy, or regulatory restrictions without requiring deep CLI intervention.

2. Managing Availability (The "Kill Switch")

There will be times—during massive cluster upgrades, critical security audits, or backend storage migrations—when you need to temporarily close the showroom entirely. You control visibility via the OdhDashboardConfig custom resource.

To hide the catalog from the UI:

oc patch odhdashboardconfig odh-dashboard-config \
  -n redhat-ods-applications \
  --type merge \
  -p '{"spec": {"dashboardConfig": {"disableModelCatalog": true}}}'

  • True: The AI hub → Catalog menu item disappears from the dashboard navigation menu.

  • False: The Catalog tab reappears.

3. The Troubleshooting Playbook: The 3.x Upgrade Gotcha

Even a validated platform has edge cases during major architectural shifts. If you are migrating a cluster to the 3.x architecture, this is the most critical operational "gotcha" you will encounter.

Issue: Dashboard URLs Return 404 Errors

  • Symptom: After upgrading to OpenShift AI 3.x, users complain that their bookmarks to the OpenShift AI dashboard (and by extension, the catalog) are returning 404 Page Not Found errors.

  • Root Cause: The transition to using the Gateway API in version 3.x changed the underlying URL structure. The dashboard URL subdomain changed from rhods-dashboard-redhat-ods-applications.apps.<cluster> to data-science-gateway.apps.<cluster>. Existing bookmarks using the old format will no longer function.

  • The Fix: Cluster administrators must provide the new dashboard URL (data-science-gateway.apps.<cluster>) to all users and update internal documentation. To ease the transition, administrators can deploy an nginx-based redirect solution that recreates the old route name and redirects traffic to the new gateway URL.

4. The Payoff: Business Value & ROI

You started this course with a question: "Why should we use this instead of just downloading files from the internet?"

You now have the answer. The model catalog is not just a UI; it is a Risk Control Plane.

Value Pillar The Engineering Reality

1. ROI & Cost

Hardware Mapping: By utilizing the Performance Insights and quantization matching, you fit massive models onto cheaper GPUs, preventing expensive over-provisioning.

2. Security & Compliance

Supply Chain Trust: Red Hat Validated Models are scanned and benchmarked. The 3.3 Allow/Disallow UI capabilities give you the ultimate veto power over what enters your environment.

3. Performance

Hardware Validated: You don’t waste weeks discovering a model is incompatible with your accelerator. If it’s in the catalog with a profile (including IBM Spyre and AMD), it runs.

4. Speed

ModelCars: OCI-based delivery means scaling out replicas takes seconds (cached layers) rather than minutes (re-downloading massive files).
