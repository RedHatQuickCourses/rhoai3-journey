Architecture & Ecosystem: Lessons from the Trenches (OpenShift AI 3.3)

We set out to build a Models-as-a-Service (MaaS) deployment using a GitOps approach. We expected a straightforward path: deploy a model, expose it, and manage access. Instead, we hit several roadblocks that forced us to look under the hood.

What we discovered fundamentally changed how we view the platform.

In OpenShift AI 3.3, MaaS is not a standalone application; it is an ecosystem. If you just click "Deploy" in the Gen AI Studio UI without understanding the backend foundation, the platform’s fail-safes will silently halt your deployment.

Here is what we learned the hard way about the "Invisible Dependency Chain," and the exact architecture you need to understand to build this factory for your customers.

1. The Invisible Dependency Chain

To make MaaS work in a scalable, GitOps-driven environment, you are actually orchestrating a stack of five distinct technologies. The inference request flows from the top down, but the infrastructure must be built from the bottom up.

  • The Network Front Door: Kubernetes Gateway API

    • What it does: Replaces standard OpenShift Routes to handle advanced HTTP/REST traffic routing into the cluster.

    • The Catch: The OpenShift AI Operator looks for a specific Gateway resource (maas-default-gateway). If it doesn’t exist, it halts the MaaS API deployment to prevent unroutable services.

  • The Governance Engine: Red Hat Connectivity Link (Kuadrant)

    • What it does: Provides the API key authentication (Authorino) and rate-limiting (Limitador) necessary to protect expensive GPU resources from abuse.

    • The Catch: The RHOAI Operator refuses to generate the maas-parameters configuration if Kuadrant is missing. It acts as a fail-safe to prevent insecure models from being exposed.

  • The Resource Bank: Kueue Operator

    • What it does: Manages quotas and queues for distributed workloads, ensuring fair sharing of GPU and CPU resources across namespaces.

    • The Catch: Distributed inference must ask Kueue for permission to use a GPU via a ClusterQueue. If Kueue isn’t configured, the deployment is rejected immediately with scheduling errors.

  • The Pod Orchestrator: LeaderWorkerSet (LWS) Operator

    • What it does: Groups pods into a "Leader" (handling API routing) and "Workers" (handling the GPU execution math).

    • The Catch: KServe requires LWS to deploy the distributed runtime. Even on a single-node cluster, KServe will fail with a ReconcileMultiNodeWorkloadError if the LWS Custom Resource Definitions (CRDs) are missing.

  • The Execution Engine: KServe & llm-d

    • What it does: The actual inference server that loads the model into vLLM and serves the tokens.

    • The Catch: This sits at the very top of the stack. It relies entirely on the Gateway for traffic, Kuadrant for security, Kueue for GPUs, and LWS for pod management.

2. The MaaS Vocabulary

When designing this architecture with customers, you must speak the language of the 3.3 release. These are the core concepts that separate MaaS from legacy model serving:

Distributed Inference Server (llm-d)

The mandatory orchestrator for MaaS. It acts as an intelligent router, receiving requests and distributing them efficiently across multiple backend vLLM worker engines. It enables cache-aware scheduling and separates the "prefill" phase from the "decode" phase to eliminate latency bottlenecks.

Service Tiers & Tokens

The business-logic layer. You define Tiers (e.g., Free, Premium) that map directly to OpenShift User Groups. These tiers dictate the strict request rates and token limits enforced by Kuadrant. Crucially, Kuadrant counts aggregate tokens, not just HTTP hits, preventing users from exhausting GPUs with massive context windows.

Hardware Efficiency (Time-Slicing & RoCE)

MaaS is designed to maximize hardware ROI. Time-Slicing allows multiple llm-d workloads to share a single physical GPU by dividing processing time. For massive models (like Llama 3 70B) that must span multiple physical nodes, RoCE (RDMA over Converged Ethernet) allows GPUs on different servers to communicate directly with sub-microsecond latency, bypassing the CPU entirely.

3. The Request Lifecycle: Tying it Together

To troubleshoot or engineer this system, you must understand the exact flow of a request through the dependency chain we just outlined:

  1. Client sends an OpenAI-compatible POST request (/v1/chat/completions) using their generated MaaS API Token.

  2. Gateway API receives the request at the edge of the OpenShift cluster.

  3. Kuadrant (Authorino) intercepts the traffic and validates the API token against the defined Service Tiers.

  4. Kuadrant (Limitador) checks the token consumption math. (If the user’s quota is exceeded, it immediately returns a 429 Error, saving GPU cycles).

  5. llm-d Orchestrator receives the validated request. It checks its cache map and routes the prompt to the most optimal worker.

  6. vLLM Worker executes the forward pass (leveraging RoCE if spanning multiple nodes) and streams tokens back to llm-d.

  7. Limitador asynchronously updates the user’s token usage in the backend database for accurate chargeback monitoring.

  8. Client receives the streamed response.
