Building the Foundation: The Platform Engineer Lab (OpenShift AI 3.3)

The GitOps Disclaimer

OpenShift AI 3.3 introduces the "Gen AI Studio" dashboard for creating Tiers, deploying models, and generating API keys visually. While the UI provides a great demo experience, industrialized enterprise customers rely on Infrastructure-as-Code.

In this lab, we are going to intentionally bypass the UI. You will use the OpenShift CLI (oc) to deploy the underlying Custom Resources (CRs) that the UI generates behind the scenes. This equips you with the exact YAML manifests you need to build a customer’s GitOps pipeline (e.g., using Red Hat OpenShift GitOps / ArgoCD).

Theory is over. We know the architecture. Now, let’s roll up our sleeves and build the factory.

In this lab, you are acting as the customer’s Lead Platform Engineer. Your job is to satisfy the "Invisible Dependency Chain" we discussed in the previous module, activate the MaaS platform, and deploy your first governed generative AI model strictly through code.

Phase 1: Pre-Flight & Operator Verification

Before we apply any configuration, ensure your cluster has the necessary operators installed from OperatorHub. If any of these are missing, the deployment will silently fail.

  1. Red Hat OpenShift Service Mesh (Provides the underlying Istio networking).

  2. Red Hat Connectivity Link (v1.2+ for Kuadrant governance). 3 . Red Hat OpenShift AI (v3.0+).

  3. LeaderWorkerSet Operator (Mandatory for llm-d orchestration).

Phase 2: Infrastructure Foundation

The OpenShift AI Operator is a strict gatekeeper. It will not provision the MaaS API unless the foundational network routing and security engines are actively running on the cluster.

1. Initialize the Governance Engine (Kuadrant)

We must tell the Connectivity Link operator to spin up Authorino and Limitador so MaaS has a place to enforce its token quotas.

cat <<EOF | oc apply -f -
apiVersion: kuadrant.io/v1beta1
kind: Kuadrant
metadata:
  name: kuadrant
  namespace: kuadrant-system
EOF

2. Create the MaaS Gateway (The Front Door)

KServe requires a specific Gateway resource to exist before it will route external traffic to your models.

cat <<EOF | oc apply -f -
apiVersion: gateway.networking.k8s.io/v1beta1
kind: Gateway
metadata:
  name: maas-default-gateway
  namespace: openshift-ingress
spec:
  gatewayClassName: openshift-gateway-controller
  listeners:
  - name: http
    port: 80
    protocol: HTTP
    allowedRoutes:
      namespaces:
        from: All
EOF

Phase 3: Platform Activation

With the infrastructure in place, we instruct the OpenShift AI operator to turn on the MaaS features.

1. Enable the MaaS Backend

Patch the DataScienceCluster to set the modelsAsService management state to Managed. This prompts the operator to build the maas-api pods and default RBAC roles.

oc patch datasciencecluster default-dsc --type=merge -p '{
  "spec": {
    "components": {
      "kserve": {
        "managementState": "Managed",
        "modelsAsService": {
          "managementState": "Managed"
        }
      }
    }
  }
}'

2. Enable the Dashboard (Gen AI Studio)

Even though we are deploying via GitOps, the AI Administrators will still need the UI to manage tokens. Turn it on:

oc patch odhdashboardconfig odh-dashboard-config -n redhat-ods-applications --type=merge -p '{
  "spec": {
    "dashboardConfig": {
      "modelAsService": true,
      "genAiStudio": true
    }
  }
}'

Phase 4: Deploying the Model (The GitOps Way)

Now we deploy the model using the llm-d runtime. When the UI does this, it creates an LLMInferenceService. Here is the exact YAML required to do it declaratively.

1. Prepare the Target Project

Create a namespace and inject it into the service mesh.

oc new-project maas-production
oc label namespace maas-production istio-injection=enabled --overwrite

2. Apply the LLMInferenceService

This manifest tells KServe to use llm-d, requests 1 GPU, and critically, applies the maas-endpoint: "true" annotation which tells Kuadrant to protect this route.

cat <<EOF | oc apply -f -
apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: qwen
  namespace: my-api-lab
  annotations:
    opendatahub.io/hardware-profile-name: gpu-profile
    opendatahub.io/hardware-profile-namespace: redhat-ods-applications
    opendatahub.io/model-type: generative
    openshift.io/display-name: qwen
    security.opendatahub.io/enable-auth: 'false'
spec:
  replicas: 1
  model:
    uri: hf://Qwen/Qwen3-0.6B
    name: Qwen/Qwen3-0.6B
  router:
    scheduler:
      template:
        containers:
          - name: main
            env:
              - name: TOKENIZER_CACHE_DIR
                value: /tmp/tokenizer-cache
              - name: HF_HOME
                value: /tmp/tokenizer-cache
              - name: TRANSFORMERS_CACHE
                value: /tmp/tokenizer-cache
              - name: XDG_CACHE_HOME
                value: /tmp
            args:
              - --pool-group
              - inference.networking.x-k8s.io
              - '--pool-name'
              - '{{ ChildName .ObjectMeta.Name `-inference-pool` }}'
              - '--pool-namespace'
              - '{{ .ObjectMeta.Namespace }}'
              - '--zap-encoder'
              - json
              - '--grpc-port'
              - '9002'
              - '--grpc-health-port'
              - '9003'
              - '--secure-serving'
              - '--model-server-metrics-scheme'
              - https
              - '--config-text'
              - |
                apiVersion: inference.networking.x-k8s.io/v1alpha1
                kind: EndpointPickerConfig
                plugins:
                - type: single-profile-handler
                - type: queue-scorer
                - type: kv-cache-utilization-scorer
                - type: prefix-cache-scorer
                schedulingProfiles:
                - name: default
                  plugins:
                  - pluginRef: queue-scorer
                    weight: 2
                  - pluginRef: kv-cache-utilization-scorer
                    weight: 2
                  - pluginRef: prefix-cache-scorer
                    weight: 3
            volumeMounts:
              - name: tokenizer-cache
                mountPath: /tmp/tokenizer-cache
              - name: cachi2-cache
                mountPath: /cachi2
        volumes:
          - name: tokenizer-cache
            emptyDir: {}
          - name: cachi2-cache
            emptyDir: {}
    route: { }
    gateway: { }
  template:
    tolerations:
      - key: "nvidia.com/gpu"
        operator: "Exists"
        effect: "NoSchedule"
    containers:
      - name: main
        env:
          - name: VLLM_ADDITIONAL_ARGS
            value: "--disable-uvicorn-access-log --max-model-len=16000"
        resources:
          limits:
            cpu: '1'
            memory: 8Gi
            nvidia.com/gpu: "1"
          requests:
            cpu: '1'
            memory: 8Gi
            nvidia.com/gpu: "1"
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
            scheme: HTTPS
          initialDelaySeconds: 120
          periodSeconds: 30
          timeoutSeconds: 30
          failureThreshold: 5
EOF

3. Map Users to Service Tiers (RBAC)

MaaS uses Kubernetes RBAC to determine which Service Tiers a user can access. To grant an application (or user) access to your model, you must bind them to the appropriate viewer role.

# Create a ServiceAccount for the application that will consume the AI
oc create serviceaccount ai-app-bot -n maas-production

# Bind it to the viewer role to grant it access to the MaaS endpoint
oc create rolebinding bot-model-access \
  --role=llm-inference-viewer \
  --serviceaccount=maas-production:ai-app-bot \
  -n maas-production
