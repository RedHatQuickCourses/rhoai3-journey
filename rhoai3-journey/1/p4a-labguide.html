Deploying Distributed Inference

Role: Platform Engineer
Objective: Deploy a cache-aware inference service, expose it via the Gateway, and verify intelligent routing.

Deploy using Distributed Inference with llm-d

We will deploy a super small Qwen/Qwen3-0.6B model for quick loading using the LLMInferenceService CR. This resource tells the operator to spin up the vLLM pods and the Inference Scheduler.

Step 1: Create the Project
oc new-project my-ai-lab
Step 2: Apply the LLMInferenceService

Save the following as llm-inference-service.yaml and apply it.

apiVersion: serving.kserve.io/v1alpha1
kind: LLMInferenceService
metadata:
  name: qwen
  namespace: my-api-lab
  annotations:
    opendatahub.io/hardware-profile-name: gpu-profile
    opendatahub.io/hardware-profile-namespace: redhat-ods-applications
    opendatahub.io/model-type: generative
    openshift.io/display-name: qwen
    security.opendatahub.io/enable-auth: 'false'
spec:
  replicas: 1
  model:
    uri: hf://Qwen/Qwen3-0.6B
    name: Qwen/Qwen3-0.6B
  router:
    scheduler:
      template:
        containers:
          - name: main
            env:
              - name: TOKENIZER_CACHE_DIR
                value: /tmp/tokenizer-cache
              - name: HF_HOME
                value: /tmp/tokenizer-cache
              - name: TRANSFORMERS_CACHE
                value: /tmp/tokenizer-cache
              - name: XDG_CACHE_HOME
                value: /tmp
            args:
              - --pool-group
              - inference.networking.x-k8s.io
              - '--pool-name'
              - '{{ ChildName .ObjectMeta.Name `-inference-pool` }}'
              - '--pool-namespace'
              - '{{ .ObjectMeta.Namespace }}'
              - '--zap-encoder'
              - json
              - '--grpc-port'
              - '9002'
              - '--grpc-health-port'
              - '9003'
              - '--secure-serving'
              - '--model-server-metrics-scheme'
              - https
              - '--config-text'
              - |
                apiVersion: inference.networking.x-k8s.io/v1alpha1
                kind: EndpointPickerConfig
                plugins:
                - type: single-profile-handler
                - type: queue-scorer
                - type: kv-cache-utilization-scorer
                - type: prefix-cache-scorer
                schedulingProfiles:
                - name: default
                  plugins:
                  - pluginRef: queue-scorer
                    weight: 2
                  - pluginRef: kv-cache-utilization-scorer
                    weight: 2
                  - pluginRef: prefix-cache-scorer
                    weight: 3
            volumeMounts:
              - name: tokenizer-cache
                mountPath: /tmp/tokenizer-cache
              - name: cachi2-cache
                mountPath: /cachi2
        volumes:
          - name: tokenizer-cache
            emptyDir: {}
          - name: cachi2-cache
            emptyDir: {}
    route: { }
    gateway: { }
  template:
    tolerations:
      - key: "nvidia.com/gpu"
        operator: "Exists"
        effect: "NoSchedule"
    containers:
      - name: main
        env:
          - name: VLLM_ADDITIONAL_ARGS
            value: "--disable-uvicorn-access-log --max-model-len=16000"
        resources:
          limits:
            cpu: '1'
            memory: 8Gi
            nvidia.com/gpu: "1"
          requests:
            cpu: '1'
            memory: 8Gi
            nvidia.com/gpu: "1"
        livenessProbe:
          httpGet:
            path: /health
            port: 8000
            scheme: HTTPS
          initialDelaySeconds: 120
          periodSeconds: 30
          timeoutSeconds: 30
          failureThreshold: 5
oc apply -f llm-inference-service.yaml
Status Check

The status may briefly show Failed in the dashboard while images pull. Check the pods: oc get pods -n my-ai-lab Wait for model and inference kserve pods to be Running.

Phase 2: Expose and Secure (The Gateway)

In OpenShift AI 3.0, you must explicitly configure the route if the UI didn’t auto-associate it.

Step 1: Verify the HTTPRoute
oc get httproute -n my-ai-lab

Ensure the route points to the openshift-ai-inference gateway.

Step 3: Get the Inference URL
export INFERENCE_URL=$(oc get gateway openshift-ai-inference -n openshift-ingress -o jsonpath='{.status.addresses[0].value}')
echo "Targeting: http://$INFERENCE_URL/my-ai-lab/qwen/v1"

Phase 3: Verify Intelligent Routing (The Test)

We will prove that the scheduler is routing based on cache affinity.

Step 1: The "Cold" Request (Baseline)

Send a request with a unique system prompt. Measure the total time.

time curl -k -X POST "http://$INFERENCE_URL/my-ai-lab/qwen/v1/chat/completions" \
  -H "{}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "Qwen/Qwen3-0.6B",
    "messages": [
      {"role": "system", "content": "You are a helpful assistant talking about OpenShift AI."},
      {"role": "user", "content": "Explain Kubernetes in 50 words."}
    ]
  }'
Step 2: The "Warm" Request (Cache Hit)

Send the exact same request immediately. The Scheduler should recognize the prefix hash and route to the same pod.

# Run the exact same command again.
# Compare the 'real' time. The second request should be significantly faster (lower TTFT).

Phase 4: Observability (Golden Signals)

Validate your test using the Dashboard metrics.

  1. Navigate to OpenShift → Observe → Dashboards.

    • Networking/Ingress

    • Prometheus

    • NVIDIA DCGM Exporter Dashboard

  2. Look for:

    • Request Count: You should see your 2 spikes.

    • Avg Response Time: The second spike should be lower than the first.
