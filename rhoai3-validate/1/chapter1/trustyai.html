Lab: The Accuracy Exam (TrustyAI)

Intelligence is measurable. Don’t guess.

In the previous course, you deployed granite-4-micro. It responds to curl requests. But does it actually know anything? Deployment proves the pipes work. Evaluation proves the water is clean.

In this module, you will use Red Hat TrustyAI to administer two exams: 1. The Standardized Test: MMLU (Massive Multitask Language Understanding) for general reasoning. 2. The Custom Exam: A domain-specific test you define yourself using task.yaml.

The Architecture: LMEvalJob

TrustyAI introduces a Kubernetes Custom Resource called LMEvalJob. When you submit this job, the TrustyAI Operator: 1. Spins up a driver pod. 2. Downloads the exam questions (datasets). 3. Connects to your model (via local-completions API). 4. Grades the answers and stores the report.

Phase 1: The Pre-Flight Check (Admin Task)

Decision Point: Online vs. Air-Gapped

By default, RHOAI blocks evaluation jobs from accessing the internet. * The Secure Path: Manually download datasets to a PVC (Air-Gapped). * The Fast Path: Allow jobs to download datasets from Hugging Face directly.

For this lab, we will use the Fast Path.

As a cluster administrator, you must explicitly enable "Online Mode" for the cluster.

  1. Check your DataScienceCluster config:

oc get datasciencecluster -n redhat-ods-operator -o yaml | grep permitOnline

  1. Enable Access (If set to deny):

oc patch datasciencecluster default-dsc -n redhat-ods-operator --type merge \
  -p '{"spec":{"components":{"trustyai":{"eval":{"lmeval":{"permitOnline":"allow"}}}}}}'

Why? This allows the LMEvalJob to fetch datasets dynamically.

Phase 2: The Standardized Test (MMLU)

First, let’s establish a baseline using a public benchmark.

Create eval-granite-mmlu.yaml
apiVersion: trustyai.opendatahub.io/v1alpha1
kind: LMEvalJob
metadata:
  name: eval-granite-mmlu
  namespace: model-deploy-lab # Same namespace as your model
spec:
  allowOnline: true # Requires the Admin setting above
  model: local-completions # Use the OpenAI-compatible API
  batchSize: 1
  taskList:
    taskNames:
      # We select a tiny subset of MMLU for speed.
      - mmlu_abstract_algebra
  modelArgs:
    - name: model
      value: granite-4-micro
    - name: base_url
      # The internal DNS of your KServe InferenceService
      # Format: http://<isvc-name>-predictor.<namespace>.svc:8080/v1/completions
      value: http://granite-4-micro-predictor.model-deploy-lab.svc:8080/v1/completions
    - name: tokenizer
      value: ibm-granite/granite-3.0-2b-instruct
  outputs:
    pvcManaged:
      size: 1Gi # Stores the results in a temporary PVC

  1. Submit the Exam:

oc apply -f eval-granite-mmlu.yaml

Phase 3: The Custom Exam (Domain Specific)

Standard benchmarks don’t test your data. Let’s create a custom "OpenShift Trivia" exam.

Step 3.1: Define the Exam Material

We need two files: the questions (data.jsonl) and the instructions (task.yaml).

  1. Create the Data (data.jsonl):

{"question": "What is the command-line tool for OpenShift?", "choices": ["kubectl", "oc", "docker", "podman"], "answer": 1}
{"question": "What resource provides a DNS-resolvable URL?", "choices": ["Route", "Ingress", "Pod", "Build"], "answer": 0}

  1. Create the Instructions (task.yaml):

task: custom-lmeval-task
dataset_path: json
dataset_kwargs:
  data_files: "/opt/app-root/src/my_tasks/data.jsonl" # Must match mount path below
doc_to_choice: choices
doc_to_text: "Question: {{question}}\n\nChoices:\n{% for choice in choices %}- ({{'ABCD'[loop.index0]}}) {{choice}}\n{% endfor %}\nAnswer:"
doc_to_target: answer
output_type: multiple_choice
metric_list:
  - metric: acc
    aggregation: mean
    higher_is_better: true
metadata:
  version: 1.0

  1. Bundle into a ConfigMap:

oc create configmap custom-lmeval-benchmark-files \
  --from-file=task.yaml=task.yaml \
  --from-file=data.jsonl=data.jsonl \
  -n model-deploy-lab

Step 3.2: Run the Custom Job

Now we deploy a job that mounts this ConfigMap into the evaluator pod.

Create eval-custom.yaml
apiVersion: trustyai.opendatahub.io/v1alpha1
kind: LMEvalJob
metadata:
  name: eval-custom-trivia
  namespace: model-deploy-lab
spec:
  model: local-completions
  taskList:
    taskNames:
      - custom-lmeval-task # Matches task.yaml
  modelArgs:
    - name: model
      value: granite-4-micro
    - name: base_url
      value: http://granite-4-micro-predictor.model-deploy-lab.svc:8080/v1/completions
    - name: include_path # Tells LM-Eval where to find the task definition
      value: /opt/app-root/src/my_tasks
    - name: tokenizer
      value: ibm-granite/granite-3.0-2b-instruct
  pod:
    volumes:
      - name: custom-benchmark-volume
        configMap:
          name: custom-lmeval-benchmark-files
    container:
      volumeMounts:
        # Mount the Task Definition
        - name: custom-benchmark-volume
          mountPath: /opt/app-root/src/my_tasks/task.yaml
          subPath: task.yaml
          readOnly: true
        # Mount the Data
        - name: custom-benchmark-volume
          mountPath: /opt/app-root/src/my_tasks/data.jsonl
          subPath: data.jsonl
          readOnly: true
  outputs:
    pvcManaged:
      size: 1Gi

  1. Submit the Custom Exam:

oc apply -f eval-custom.yaml

Phase 4: Analyze the Results

  1. Monitor the Proctor:

oc get lmevaljob eval-custom-trivia -w

  1. Read the Report Card:

oc get lmevaljob eval-custom-trivia -o jsonpath='{.status.results}' | jq

  • Expectation: If your model knows OpenShift, the acc score should be high (e.g., 1.0 for 2 questions). If it guesses randomly (A, B, C, D), the score will be ~0.25.
