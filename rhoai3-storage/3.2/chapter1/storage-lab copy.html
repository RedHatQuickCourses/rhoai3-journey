Lab: Constructing the AI Supply Chain

Stop clicking. Start engineering.

In a mature AI Factory, you do not manually type AWS keys into a web form for every new project. You build automation that governs how data is accessed and stored.

In OpenShift AI 3.3, storage configuration dictates pipeline performance and model startup velocity. In this lab, you will act as the Lead Platform Engineer. You will replace manual configuration with a set of standardized deployment scripts that instantly provision the three pillars of RHOAI storage: Persistence (Pipeline Workspaces), Acceleration (OCI Modelcars), and Connectivity (S3/URI).

Prerequisites

  • Access to an OpenShift cluster with OpenShift AI 3.3 installed.

  • cluster-admin privileges (required to patch Persistent Volumes).

  • The oc CLI tool installed.

1. Initialize the Workspace

We have packaged the automation scripts into a Git repository.

# Clone the automation repo
git clone https://github.com/RedHatQuickCourses/rhoai3-storage.git
cd rhoai3-storage/deploy
chmod +x *.sh

# Create the project for our factory
oc new-project ai-supply-chain

Part 1: The Foundation (Pipeline Workspaces & Shared Storage)

In OpenShift AI 3.3, Persistent Volume Claims (PVCs) are no longer just scratchpads for data scientists; they are the high-speed Pipeline Workspaces that allow AI pipelines to pass massive datasets between tasks without the latency of S3.

We will provision a shared volume. Furthermore, we will apply a Safety Protocol (Retain policy) to ensure this data survives accidental namespace deletion.

Action: Provision & Protect

# 1. Provision Shared Storage (100GB, ReadWriteMany)
# Usage: ./cluster-storage.sh <project> <name> <size> <mode>
./cluster-storage.sh ai-supply-chain golden-dataset 100 rwx
Storage Class Limitations

If your cluster only supports Block Storage (e.g., AWS EBS gp3-csi), the rwx command above will leave the PVC in a Pending state. If you do not have shared file storage, run this instead to force a single-user volume: ./cluster-storage.sh ai-supply-chain golden-dataset 100 rwo

 
# 2. THE CRITICAL SAFETY STEP: Patch the Reclaim Policy to 'Retain'
# Find the Bound PV name
PV_NAME=$(oc get pvc golden-dataset -n ai-supply-chain -o jsonpath='{.spec.volumeName}')

# Patch the Persistent Volume
oc patch pv $PV_NAME -p '{"spec":{"persistentVolumeReclaimPolicy":"Retain"}}'

# Verify
oc get pv $PV_NAME

Result: You should see RECLAIM POLICY: Retain. Your pipeline workspace and datasets are now protected from accidental deletion.

Part 2: The Velocity Tier (OCI Modelcars)

Next, we enable "Instant Scaling" by creating a connection to the Red Hat Modelcar registry. This tells OpenShift AI 3.3 to use the underlying CRI-O container cache rather than downloading massive files.

Action: Connect the Accelerator

# Usage: ./modelcar.sh <project>
./modelcar.sh ai-supply-chain

Result: A Secret granite-3-8b-modelcar is created. Note that the script utilizes the required 3.3 annotation: opendatahub.io/connection-type-protocol: oci.

Part 3: The Air-Gap Simulation (The Helper Pod)

In a highly secure enterprise, your cluster might not have internet access to download models from the Hugging Face Hub. You need a way to "Side-Load" data directly into your PVC so you can use the OpenShift AI 3.3 pvc:// deployment routing.

We will use the Helper Pod Pattern to stage data onto our golden-dataset volume.

Action: Start the Loader

# 1. Spin up a temporary UBI pod with the PVC mounted
cat <<EOF | oc apply -n ai-supply-chain -f -
apiVersion: v1
kind: Pod
metadata:
  name: data-loader
spec:
  containers:
    - name: loader
      image: registry.access.redhat.com/ubi9/ubi-minimal
      command: ["/bin/sh", "-c", "sleep 3600"]
      volumeMounts:
        - mountPath: "/data"
          name: my-vol
  volumes:
    - name: my-vol
      persistentVolumeClaim:
        claimName: golden-dataset
EOF

# 2. Wait for it to be Ready
oc wait --for=condition=Ready pod/data-loader -n ai-supply-chain --timeout=120s

# 3. Simulate "Side-Loading" a Model
echo "Simulated Model Weights v1.0" > model.safetensors
oc cp model.safetensors ai-supply-chain/data-loader:/data/model.safetensors

# 4. Clean up the Helper Pod
oc delete pod data-loader -n ai-supply-chain

Result: You have successfully bypassed the internet and staged data directly onto a disconnected volume. Because of our Retain policy, deleting the pod keeps the data safe on the disk.

Part 4: The Ingestion Layer (S3 & URI)

Finally, we provide the standard connections for raw data ingestion and precise external links.

Action: Connect Data Lakes and URIs

# Connect the simulated corporate data lake
./dataconnect.sh ai-supply-chain corporate-data-lake

# Connect a precise URI to a specific external model file
./uri.sh ai-supply-chain

Part 5: The 3.3 Payoff (Deployment Routing)

You have just architected a complete storage solution via the CLI. Now, let’s see how OpenShift AI 3.3 consumes this architecture using Direct URIs.

  1. Open the OpenShift AI Dashboard.

  2. Navigate to Data Science Projects → ai-supply-chain.

  3. Under the Data connections tab, verify your connections are present and labeled correctly (S3, OCI, URI).

  4. Click the Deployments tab, and click Deploy model.

Notice how the UI expects you to route the deployment based on the architecture you just built. You can now define your deployment locations using the exact 3.3 syntax: * To use the S3 Connection: s3://private-models/my-model * To use the OCI Accelerator: oci://quay.io/redhat-ai-services/modelcar-catalog:granite-3.2-8b-instruct * To use the Air-Gapped PVC: pvc://golden-dataset/model.safetensors

The Architecture You Built

  • Safe: Your shared data cannot be accidentally deleted by Kubernetes.

  • Fast: Your models stream from OCI, leveraging node caching.

  • Secure: Your credentials are injected via the Connections API, not hardcoded.

  • Flexible: You have a workflow for loading data in air-gapped environments.
