Field Notes & Troubleshooting

Configuration is easy. Day-2 Operations are where the battle is won.

Now that you understand the physics of AI storage, Custom Connection Types, and the new OpenShift AI 3.3 Pipeline architecture, you need to maintain it under heavy load.

In a high-velocity industrial AI environment, default Kubernetes settings will could sabotage your Generative AI workloads. Based on real-world deployments and the latest 3.3 architectural shifts, here are the configurations and operational patterns that help separate a stable platform from a fragile one.

1. The Custom Connection Mandate (Preventing Rogue Data Access)

We discussed Custom Connections in earlier phases, but it remains your number one operational defense on Day 2.

  • The Shadow AI Failure: Data scientists bypassing the platform to manually hardcode database credentials or connect to untrusted external buckets directly in their Python code.

  • The Factory Fix: You must enforce the use of ConnectionType Custom Resources for all structured and unstructured data. By strictly defining the "Socket" (e.g., Corporate PostgreSQL, Amazon S3, MinIO) and injecting the "Plug" (Data Connection secrets), you ensure all data access is auditable, repeatable, and managed by GitOps.

The "Protocol" Pitfall (Connectivity)

When configuring Data Connections for S3-compatible stores (MinIO, Ceph) via the UI or GitOps, the most frequent operational error is Endpoint Formatting.

Entering minio-api.cluster.local:9000 into the dashboard connection form will fail. You must include the HTTP/HTTPS protocol scheme (e.g., http://minio-api.cluster.local:9000). The underlying Boto3 clients injected into workbenches and pipelines strictly require the full URI scheme to resolve correctly.

2. The "Timeout" Trap (OCI Modelcars)

When you use the OCI protocol to stream a massive foundation model (e.g., 70GB Granite or Llama model), the underlying container runtime (CRI-O) must pull all layers over the network before the Pod can transition to a "Ready" state.

  • The Shadow AI Failure: Knative (the serverless engine behind Model Serving) defaults to a progress-deadline of 600 seconds (10 minutes). If your network takes 11 minutes to pull the image, Knative marks the deployment as Failed and ruthlessly kills the pod—even if it was 99% complete.

  • The Factory Fix: You must explicitly override the default physics. Increase the progressDeadlineSeconds in your InferenceService (or global Knative config) to 1800s (30 mins) or higher to accommodate large LLM pulls and protect your Time-to-First-Token (TTFT) metrics.

3. The "Workspace Import" Trap (3.3 Pipelines)

OpenShift AI 3.3 introduces Ephemeral Pipeline Workspaces (PVCs) for high-speed data transit between tasks. However, you cannot directly download an OCI Modelcar into a workspace volume natively.

  • The Shadow AI Failure: If a pipeline task expects an OCI image but you only provide the oci:// URI, the pipeline task will fail because it cannot find the extracted model files on the workspace disk.

  • The Factory Fix (Side-Loading): You must explicitly define an importer component in your pipeline code to pull the OCI artifact. Then, you must use a subsequent Python component (e.g., using shutil.copytree) to physically copy the artifact from the importer into the dsl.WORKSPACE_PATH_PLACEHOLDER.

4. The "Dirty Database" Migration Error

In OpenShift AI 3.3, the ML Metadata (MLMD) server was completely removed from the Model Registry. The registry now interacts directly with your mandated external database (MySQL/MariaDB).

  • The Shadow AI Failure: If you are upgrading from a previous version, your database schema migration might fail. Your Model Registry pod will crash with the following log: error: error connecting to datastore: Dirty database version {version}. Fix and force version..

  • The Factory Fix: You must manually intervene as the database administrator to reset the "dirty" state. Exec into the database pod and run the following command:

# Reset the schema migration state to allow the Model Registry to start
mysql -u root -p"$MYSQL_ROOT_PASSWORD" -e "USE <your-db-name>; UPDATE schema_migrations SET dirty=0;"

5. The "Data Loss" Risk (PVC Reclaim Policy)

By default, standard cloud Storage Classes (like AWS gp3-csi) have a reclaimPolicy set to Delete.

  • The Shadow AI Failure: A Data Scientist accidentally deletes their project namespace. The underlying cloud volume is immediately scrubbed by the cloud provider. The multi-terabyte "Golden Dataset" or active Pipeline Workspace is gone forever.

  • The Factory Fix: For any Shared Storage (RWX) or Persistent Volume hosting critical, irreplaceable assets, you must patch the physical PV to Retain.

# Protect your Golden Dataset from accidental namespace deletion
oc patch pv <pv-name> -p '{"spec":{"persistentVolumeReclaimPolicy":"Retain"}}'

6. The "Trust" Gap (Certificates)

If you are using an on-premise Object Store (MinIO, Ceph) or an external database (Amazon RDS) secured with Self-Signed Certificates, your Data Connections will fail silently.

  • The Shadow AI Failure: Workbenches hang, pipelines fail to compile, or you see tls: failed to verify certificate: x509: certificate signed by unknown authority.

  • The Factory Fix: You must inject your OpenShift Cluster’s kube-root-ca.crt bundle into the DSCI (Data Science Cluster Initialization) resource. Platform components (Pipelines, Model Registry) will not trust your storage backend without it.

This concludes the core enablement for OpenShift AI Storage Logistics. By transitioning from Shadow AI to the Factory Model, you have secured your connections, protected your datasets, and optimized your GPU ROI.
