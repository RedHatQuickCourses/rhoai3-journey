== The Problem: The "Magic Lab" Doesn't Exist

Right now, organizations are facing a GPU crisis where expensive, limited hardware is trapped within individual teams. This leads to simultaneous idle capacity and frustrating project bottlenecks. The market often forces enterprises into a costly "token-consumption" model, renting intelligence from external providers with unpredictable costs and data risks.

We all know the pain point: there is no "magic lab" that comes pre-configured with all the new Red Hat OpenShift AI features working out-of-the-box. Figuring out how to spin up a Model Registry, load your own model, and determine exactly which operators you need is a massive bottleneck. 

== The Promise: High-Velocity Enablement

Your business mandate is to flip that narrative: to move from consuming tokens to generating them at a lower cost than public APIs. We need to treat Red Hat OpenShift AI as an on-premise industrial token factory and saturate that GPU hardware for maximum ROI.

If you have a week to sit through end-to-end training, do it. But if you are a platform engineer starting with a greenfield Red Hat OpenShift AI environment  and you need a specific piece working today to build a Proof of Concept, this guide is for you. 

We are skipping the fluff to shrink your deployment from weeks to hours. By the time you finish, you won’t just know what the buttons do ; you will have deployed and engineered the working foundation of a generative AI token-generating platform.

== The 3 Pillars of Enablement

This isn't a massive, rigid course. This approach differs from the traditional model by prioritizing business value and simplified deployment over technical depth. Every standalone module  is built on three core pillars:

* **The Business Value:** We lead with the "Why." Every module defines how a component generates ROI for the customer.
* **The Architecture:** The essential infrastructure decisions mapping hardware to specific workloads.
* **The Deployment:** Simple, copy-paste CLI and YAML instructions focusing on a specific Red Hat OpenShift AI capability's deployment and configuration.

== The Modular Approach: Building the Factory

Each phase—and its constituent modules—functions as a standalone lesson. You do not have to complete them in order! 

[cols="1,3,2"]
|===
| Phase | Focus & Outcome | Key Components

| *Phase 0:* OpenShift Operator Ecosystem
| **Installation and configuration:** Install dependent infrastructure operators and the core OpenShift AI Operator. Configure cluster policies and settings to ensure the platform is ready.
| Red Hat OpenShift AI Operator, Node Feature Discovery (NFD) Operator, NVIDIA GPU Operator, Kueue Operator (RHBOK), and more.

| *Phase 1:* Assembling the Factory
| **Configuring and connecting resources:** Establish the model inventory, connect storage solutions, and set up databases required to run the Model Registry. Abstract the infrastructure so GPUs can be shared and queued efficiently.
| Hardware Profiles, Kueue, Model Registry, Model Catalog, Data Connections, Model Storage.

| *Phase 2:* Token Production Engine
| **Single-model high-throughput serving:** Deploy your selected models for inference using vLLM. Introduce cost-optimization strategies, such as using LLM Compressor to serve lower precision models.
| vLLM, LLM Compressor, Inference Services.

| *Phase 3:* Scaling the Utility
| **Distributed Inference, MaaS, and Observability:** Move to distributed inference using _llm-d_ to handle massive workloads. Expose models via Model-as-a-Service (MaaS) API endpoints and implement observability metrics.
| llm-d, MaaS APIs, AI Gateway, Prometheus/Grafana.

| *WIP* *Phase 4:* Governance & Compound Systems
| **Locking down models and enabling Agents:** Lock down served models using TrustyAI and Guardrails to ensure safety, fairness, and compliance. This governed environment becomes the trusted launchpad for developing autonomous Agentic applications and Compound AI Systems.
| TrustyAI, Guardrails.
|===
