== 1. The Core: Red Hat OpenShift AI Operator

The Red Hat OpenShift AI Operator is the control plane that manages the lifecycle of all subsequent data science and model serving components.

=== Procedure: Apply the Operator Manifests

Execute the following block directly in your terminal to create the dedicated namespace, the `OperatorGroup`, and the `Subscription`. We are targeting the `stable-3.x` channel to ensure we receive the latest architectural updates for our factory.

[source,bash]
----
cat <<EOF | oc apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: redhat-ods-operator
---
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: rhods-operator
  namespace: redhat-ods-operator
spec:
  targetNamespaces:
  - redhat-ods-operator
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: rhods-operator
  namespace: redhat-ods-operator
spec:
  name: rhods-operator
  channel: stable-3.x
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF
----

=== Verification & Value Context

Wait a few moments for Operator Lifecycle Manager (OLM) to pull the images and start the pods. Then, verify the ClusterServiceVersion (CSV) has reached the `Succeeded` phase:

[source,bash]
----
oc get csv -n redhat-ods-operator
----

* **The GenAI Value:** This operator is the master orchestrator. While it isn't serving models yet, it is waiting for your command (via the `DataScienceCluster` resource we will configure later) to spin up the Model Catalog, the Model Registry, and the KServe engines.

== 2. The Compute Prerequisite: Node Feature Discovery (NFD)

You cannot right-size a factory if you do not understand the manufacturing requirements of your product. Similarly, you cannot schedule a massive LLM if your Kubernetes cluster is blind to the physical GPUs attached to its worker nodes. 



The NFD operator scans your nodes and applies specific hardware labels (e.g., highlighting that a node has an NVIDIA H100 or an AMD MI300X). The GPU operators rely entirely on these labels to know where to install their drivers.

=== Procedure: Install the NFD Operator

Deploy the NFD operator into its own dedicated namespace:

[source,bash]
----
cat <<EOF | oc apply -f -
apiVersion: v1
kind: Namespace
metadata:
  name: openshift-nfd
---
apiVersion: operators.coreos.com/v1
kind: OperatorGroup
metadata:
  name: openshift-nfd
  namespace: openshift-nfd
spec:
  targetNamespaces:
  - openshift-nfd
---
apiVersion: operators.coreos.com/v1alpha1
kind: Subscription
metadata:
  name: nfd
  namespace: openshift-nfd
spec:
  channel: stable
  name: nfd
  source: redhat-operators
  sourceNamespace: openshift-marketplace
EOF
----

=== Procedure: Activate the Hardware Scan

Installing the NFD operator simply deploys the controller. To actually initiate the hardware scan across your cluster, you must create the `NodeFeatureDiscovery` custom resource.

[source,bash]
----
cat <<EOF | oc apply -f -
apiVersion: nfd.openshift.io/v1
kind: NodeFeatureDiscovery
metadata:
  name: nfd-instance
  namespace: openshift-nfd
spec:
  operand:
    image: registry.redhat.io/openshift4/ose-node-feature-discovery:v4.14
  workerConfig:
    configData: |
      core:
        sleepInterval: 60s
EOF
----

=== Verification & Value Context

Check that the NFD worker pods are successfully running on your nodes:

[source,bash]
----
oc get pods -n openshift-nfd -l app=nfd-worker
----

* **The GenAI Value:** NFD is the foundation of hardware governance. By labeling your nodes, NFD allows you to subsequently deploy the NVIDIA or AMD operators and configure precise Hardware Profiles. This prevents developers from accidentally scheduling an expensive H100 GPU for a simple CPU-bound data preprocessing task.
