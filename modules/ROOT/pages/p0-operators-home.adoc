*Stop consuming tokens. Start producing them. But first, you must build the factory.*

Currently, the market often forces enterprises into a "token-consumption" model—renting intelligence from external providers, which leads to unpredictable costs and data risks. Attempting to adopt Generative AI through ad-hoc experimentation is a waste of resources. To build a secure, enterprise-grade AI system, you must flip that narrative and move from consuming tokens to generating them.

This course is **Phase 0** of the *Industrialized AI Token Platform* learning journey. The subsequent modules in this series assume a fully running, optimized OpenShift AI 3.x environment. This course teaches you exactly how to build that environment by deploying the specific Kubernetes Operators required to support your factory.

== The Dependency Matrix: Mapping Operators to the Journey

Deploying every operator in the ecosystem is an anti-pattern that wastes compute resources. A true platform engineer provisions only what the specific phase requires. 

Use this matrix to understand exactly which operators unlock which capabilities in your 4-phase learning journey:

[cols="1,2,2", options="header"]
|===
| The Journey Phase | Required Operators | What it Unlocks

| **Phase 1: Assembling the Factory** | `Red Hat OpenShift AI Operator`, `Node Feature Discovery (NFD)`, `NVIDIA/AMD GPU Operators`, `Red Hat build of Kueue`
| 
* **The Supply Chain:** Unlocks Data Connections, the Model Catalog, and the Model Registry (backed by MySQL 8+ or Postgres).
* **The Machinery:** Unlocks hardware detection, Hardware Profiles, and GPU-as-a-Service capabilities (including MIG and Kueue for fair-share scheduling) to end the GPU bottleneck.

| **Phase 2: Token Production Engine**
| `cert-manager Operator`
| 
* **Single-Model Serving:** Unlocks secure KServe `InferenceServices` using `RawDeployment` to avoid cold starts. 
* **Tuning:** Supports deploying models via vLLM with PagedAttention, allowing you to tune parameters like `--max-model-len` and `--gpu-memory-utilization` via GitOps.

| **Phase 3: Scaling the Utility**
| `Leader Worker Set`, `Connectivity Link (v1.2+)`
| 
* **Distributed Inference:** Unlocks `llm-d` to pool resources for high-throughput, massive workloads.
* **MaaS (Tech Preview):** Unlocks the invisible dependency chain (Gateway API, Kuadrant, Kueue) to provide internal Model-as-a-Service APIs with strict authentication and rate-limiting.

| **Phase 4: Governance & Compound Systems**
| `Llama Stack Operator`, `Service Mesh`, `OpenTelemetry` *(Future Additions)*
| 
* **Locking it Down:** Prepares the platform for TrustyAI and Guardrails to ensure safety, fairness, and compliance.
* **Agentic Workflows:** Establishes the governed foundation needed to support Compound AI Systems and autonomous Agentic applications.
|===

== Your Mission: Assemble the Platform

In this course, you will take on the role of the Platform Engineer to lay the groundwork for this entire journey. We will avoid local machine configurations entirely by utilizing native cloud tools.

**You will execute the following workflow:**

1. **Lab 0:** Install the OpenShift Web Terminal Operator to establish a zero-dependency Command Center.
2. **Lab 1:** Install the core OpenShift AI Operator and map your physical hardware using NFD to support Phase 1.
3. **Lab 2:** Deploy the foundational dependencies (`cert-manager` and `Connectivity Link`) required for Phase 2 serving and Phase 3 MaaS scaling.
4. **Lab 3:** Configure the `DataScienceCluster` to activate the exact components needed for your token factory.
